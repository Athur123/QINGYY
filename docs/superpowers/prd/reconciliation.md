# 社保对账复核 PRD

## Problem Statement

青阳云HRO系统根据员工参保异动自动生成应缴费用（汇缴/补缴/调基补差），但补缴和补差类型缺少应缴月份信息。社保局台账（Excel导入）记录了应缴月份，但不区分费用类型。财务人员需要将系统侧费用与台账侧账单逐笔核对，确保缴费金额一致，并将核对结果归档用于后续付款申请。

核心矛盾：**系统有费用类型、无应缴月份；台账有应缴月份、无费用类型**。双方共同拥有身份证号、险种、金额。同一员工同一险种可能产生多笔费用，金额相同的记录无法自动区分。

当前流程依赖人工逐行比对 Excel，效率低、易出错，缺乏批量操作和归档追溯能力。

## Solution

提供两层结构的对账复核系统：

1. **汇总列表页**：按参保规则维度展示对账概览，支持批量导入台账、批量对账、批量归档
2. **明细核对页**：系统侧/台账侧 Tab 分离展示，按身份证+险种+月份分组进行精确金额配对，支持人工确认、强制核对、差异处理

核对完成后将已匹配记录归档为批次，同一账单月份支持多次归档（增量数据场景），归档后可按批次筛选和追溯。

## User Stories

### 台账导入
1. 作为财务人员，我想导入社保局台账 Excel，以便与系统侧费用进行比对
2. 作为财务人员，我想在导入时选择参保主体、参保规则和台账月份，以便数据正确归属
3. 作为财务人员，我希望能看到导入预览（成功条数、错误条数、错误详情），以便在确认前修正问题
4. 作为财务人员，我希望系统对身份证格式、险种名称、金额格式自动标准化，减少手动处理
5. 作为财务人员，当我重复导入同一参保主体+规则+月份的台账时，希望系统提示是否覆盖已有数据

### 对账匹配
6. 作为财务人员，我想点击"开始对账"后自动执行金额配对，系统侧和台账侧按身份证+险种+月份分组，同金额自动匹配，双向回填信息
7. 作为财务人员，当同一分组内金额有多笔匹配时，我希望系统标记为"待确认"并提供智能推荐配对
8. 作为财务人员，我可以在配对抽屉中手动调整系统侧和台账侧的映射关系，然后确认配对
9. 作为财务人员，当系统侧费用在台账中无匹配时，我希望系统区分处理：汇缴标记为"差异"（系统多），补缴/调基补差标记为"未匹配"（可能在后续月份匹配）
10. 作为财务人员，当台账侧记录在系统中无对应时，我希望系统标记为"差异"（台账多）
11. 作为财务人员，当系统侧金额与台账侧金额不一致时，我希望系统标记为"差异"并提供差额明细

### 强制核对
12. 作为财务人员，对于已确认为正确的差异或未匹配记录，我想要"强制核对"直接标记为已核对，跳过台账匹配
13. 作为财务人员，台账侧记录不允许强制核对——必须有对应的系统侧账单才能匹配
14. 作为财务人员，强制核对后我可以取消操作，恢复到原始的差异/未匹配状态

### 明细核对
15. 作为财务人员，我想在明细页通过 Tab 切换查看系统侧账单和台账侧账单，两侧各自独立筛选
16. 作为财务人员，我想按核对状态（全部/已核对/待确认/未匹配/差异）和费用类型（全部/汇缴/补缴/调基补差）筛选记录
17. 作为财务人员，已核对记录应在系统侧和台账侧同时展示，并带有跨引用标记（↔ Txxx / ↔ Sxxx）
18. 作为财务人员，我想对系统侧待确认记录批量确认核对，对差异记录批量操作
19. 作为财务人员，我想点击差异记录的详情按钮查看差异分析和可能的处理方案

### 归档批次
20. 作为财务人员，当所有核对完成且无差异时，我想点击"归档结果"将已核对记录归档
21. 作为财务人员，同一账单月份可能有新增费用（新员工入保、补缴等），我希望能多次归档，每次生成新批次
22. 作为财务人员，我想查看历史归档批次列表（批次号、记录数、金额、归档时间）
23. 作为财务人员，我想点击某个批次查看该批次包含的所有明细记录
24. 作为财务人员，我想通过归档批次下拉筛选器快速过滤某批次的记录
25. 作为财务人员，已归档记录不可取消核对，防止误操作
26. 作为财务人员，当存在未处理的待确认或差异记录时，归档前应有确认提示

### 汇总列表
27. 作为财务人员，我想在汇总列表页看到所有参保规则的对账概览（系统侧已核对/差异/已归档，台账侧总计/已核对/差异/待确认）
28. 作为财务人员，汇总列表按地区+参保主体分组展示，可展开查看各规则明细
29. 作为财务人员，我可以在汇总页勾选规则进行批量对账、批量归档
30. 作为财务人员，我可以在汇总页导入一份总台账文件，系统自动按规则拆分
31. 作为财务人员，已归档的规则在汇总列表中显示为灰色，不可重复归档

## Implementation Decisions

### 数据模型

**系统侧费用记录核心字段**：matchStatus（UNMATCHED/MATCHED/PENDING/DIFF/PAID）、matchedLedgerId、diffType（system_more/ledger_more/amount_mismatch）、diffAmount、payableMonth、archived、archiveBatchId。费用类型（feeType）包含汇缴/补缴/调基补差。

**台账侧记录**：从 Excel 导入，包含姓名、身份证、险种、应缴月份、金额。匹配后回填 feeTypeInferred 和 matchedSystemId。

**归档批次**：{ id, label, billingMonth, createdAt, recordCount, totalAmount }。记录通过 archiveBatchId 归属到批次。

### 核对状态机

```
UNMATCHED —→ MATCHED（自动配对 / 确认配对 / 强制核对）
UNMATCHED —→ PENDING（同金额多笔）
UNMATCHED —→ DIFF（金额无匹配或金额不一致）
PENDING —→ MATCHED（确认核对后）
DIFF —→ MATCHED（强制核对，仅系统侧）
MATCHED —→ PENDING（取消核对，已归档不可取消）
MATCHED —→ archived=true（归档，创建批次）
MATCHED(archived=true) —→ PAID（付款后，matchStatus 变更为 PAID）
```

PAID 状态已在设计中定义，付款功能待后续实现。archived 是独立布尔标记，非 matchStatus 值。

### 匹配算法

按 `身份证|险种|应缴月份` 分组 → 组内按金额精确配对：
- **1:1 金额一致** → 自动 MATCHED，系统回填 payableMonth，台账回填 feeTypeInferred
- **多笔同金额** → PENDING，进入配对抽屉人工确认
- **系统有台账无** → 汇缴标记 DIFF(system_more)，补缴/调基补差标记 UNMATCHED
- **台账有系统无** → DIFF(ledger_more)
- **金额不一致** → DIFF(amount_mismatch)

特殊标记 `forcePending`（原型演示用）：即使金额完全匹配也标记为 PENDING，模拟需人工审核场景。后端对应字段 `requires_manual_review`。

### 险种映射

支持别名标准化：'基本养老保险'→'养老'、'基本医疗保险'→'医疗' 等，含大额医疗。无法识别的险种标记为"未知"留待人工处理。

### 两层页面结构

**汇总列表页** → 点击规则进入 **明细核对页**（通过 URL 参数 ?ruleName=X&month=Y 传递上下文）。明细页也可以通过 ?groupId=region|insuranceSubject 查看整个主体下的所有规则账单。

### 归档批次设计

每次归档操作将当前所有 MATCHED 且 archived=false 的记录打包为一个新批次。批次编号自动递增（第1批/第2批/...）。同一账单月份可多次归档。归档后的记录 matchStatus 保持 MATCHED，archived=true，操作列从"取消"变为"详情"。已归档记录不可取消核对。

归档批次历史通过右侧抽屉查看，点击"查看此批次"关闭抽屉并设置筛选下拉为该批次，表格自动过滤。

### 强制核对规则

仅系统侧 DIFF/UNMATCHED 记录可强制核对。操作后记录标记 forceMatched=true，matchedLedgerId=null，payableMonth=billingMonth，状态变为 MATCHED。取消强制核对时恢复到原始 DIFF/UNMATCHED 状态。台账侧不允许强制核对。

### 交互模式

- **Tab 切换**：系统侧/台账侧各自独立筛选，Tab 栏 sticky 吸顶
- **筛选栏**：核对状态（带金额提示）+ 费用类型 + 归档批次下拉
- **右侧抽屉**：配对确认抽屉、差异详情抽屉、归档批次抽屉
- **批量操作**：各 Tab 独立的全选/确认/取消/强制核对逻辑
- **确认弹窗**：强制核对确认、归档确认（DIFF/PENDING 未处理时提示）

## Testing Decisions

### 测试策略

由于原型为纯前端静态 HTML，测试以浏览器交互验证为主：

1. **功能测试**：Playwright MCP 浏览器自动化，验证完整用户流程
2. **状态机测试**：覆盖所有状态转换路径（UNMATCHED→MATCHED→PENDING、DIFF→MATCHED、MATCHED→archived、取消核对等边界条件）
3. **UI 验证**：截图对比确认页面渲染正确、CSS 无 404 错误
4. **数据一致性**：归档前后记录数核对、批次金额聚合校验

### 关键测试场景

- 台账导入：正常导入、空文件、格式错误、判重覆盖
- 对账匹配：全部自动匹配、部分待确认、全部差异
- 批量操作：全选→批量确认、全选→批量强制核对
- 归档：正常归档（无差异）、有差异时确认归档、多次归档同一月份
- 强制核对：单行+批量、取消恢复、台账侧按钮不存在
- 边界：空批次归档 toast、已归档取消核对被拦截

## Out of Scope

- 后端 API / 数据库实现

- 多语言支持
- 与社保局系统的实时对接
