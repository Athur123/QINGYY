## 2. 业务背景与问题定义

青阳云 HRO 系统根据员工参保异动自动生成应缴费用，费用类型包含：

- 汇缴
- 补缴
- 调基补差

当前业务上的核心问题是：

- 系统侧知道费用类型，但补缴和调基补差初始不知道应缴月份
- 台账侧知道应缴月份，但不区分费用类型
- 双方共同拥有的稳定匹配信息主要是：身份证、险种、金额

在一个应缴月份内，同一员工同一险种可能产生多笔费用：

- 1 笔汇缴
- N 笔补缴
- M 笔调基补差

台账中也可能存在多笔同金额记录，因此无法仅依靠金额自动判断类型归属。这会导致人工核对成本高、漏核和错核风险大，也影响后续归档与付款。

## 3. 产品目标

### 3.1 业务目标

建设一个可操作、可追踪、可归档、可付款衔接的对账复核模块，帮助运营人员完成：

- 导入台账
- 自动匹配
- 人工确认待核记录
- 处理差异
- 归档结果
- 从归档结果发起付款

### 3.2 产品目标

- 降低逐笔人工比对成本
- 将汇缴、补缴、调基补差纳入统一核对流程
- 明确系统侧与台账侧的状态流转
- 保证核对结果可归档、可筛选、可追溯
- 为测试提供完整可验收口径

### 3.3 成功标准

满足以下条件即可认为该模块达到投入使用标准：

- 台账可按规范导入并完成预览校验
- 系统可基于主规则完成自动匹配
- 待确认和差异记录可通过页面完成处理
- 已核对记录可归档为批次
- 已归档批次可发起付款并进入已付款状态
- 汇总页与明细页能在相同业务口径下联动展示

## 5. 角色与使用场景

### 5.1 主要使用角色

**社保运营 / 结算运营**

核心任务：

- 导入指定账单月份的台账文件
- 启动对账
- 处理待确认和差异记录
- 对已核对结果归档
- 从归档批次发起付款

### 5.2 使用场景

典型使用流程如下：

1. 用户进入汇总列表页，在当前账单月份上下文下查看各参保主体 / 规则的对账概况
2. 用户进入某规则或某主体的明细核对页
3. 用户在当前账单月份上下文下导入台账，台账月份自动继承且不可修改
4. 用户点击开始对账，系统自动产出 MATCHED / PENDING / DIFF / UNMATCHED 结果
5. 用户处理待确认和差异记录
6. 用户对已核对结果执行归档
7. 用户从归档批次发起付款，记录变更为已付款

## 6. 核心业务概念与术语

### 6.1 核心概念

| 概念 | 定义 |
|------|------|
| 账单月份 | 当前页面上下文中的对账账单月份 |
| 应缴月份 | 作为正式匹配输入的月份字段 |
| 费款所属期 | 费用实际归属月份，不一定等于应缴月份 |
| 汇缴 | 常规应缴费用，初始就有应缴月份 |
| 补缴 | 历史漏缴情形，对账前应缴月份可能为空 |
| 调基补差 | 因基数调整或政策调整形成的补差费用，对账前应缴月份可能为空 |
| 系统侧记录 | 系统基于参保异动生成的费用记录 |
| 台账侧记录 | 由 Excel 导入的外部台账记录 |
| 归档批次 | 将已核对记录打包后的业务批次 |

### 6.2 关键字段

| 字段 | 说明 |
|------|------|
| `billingMonth` | 当前对账账单月份（页面上下文） |
| `payableMonth` | 应缴月份，主匹配月份字段 |
| `feePeriod` | 费款所属期 |
| `feeType` | 系统侧费用类型：汇缴 / 补缴 / 调基补差 |
| `feeTypeInferred` | 台账侧匹配后回填的推断费用类型 |
| `matchStatus` | 核对状态 |
| `archived` | 是否归档，独立于 `matchStatus` |
| `archiveBatchId` | 归属归档批次 ID |

## 9. 汇总列表页需求

### 9.1 页面目标

汇总列表页用于从规则维度和参保主体维度总览对账状态，支持批量导入、批量对账、批量归档和跳转到明细页。

### 9.2 顶部区域

顶部应包含：

- 账单月份上下文选择
- 地区筛选
- 操作按钮：
  - 刷新
  - 批量导出
  - 批量对账
  - 批量归档
  - 导入台账
  - 标注（用于当前原型说明）

### 9.3 信息栏

信息栏显示：

- 规则总数
- 参保主体数
- 已导入数
- 已归档数

账单月份仅作为当前对账上下文，不过滤主体 / 规则清单；切换账单月份后，规则总数和参保主体数应保持不变。

### 9.4 表格结构

表格采用二级表头：

```text
系统侧明细（已核对 / 差异 / 已归档） | 台账侧明细（总计 / 已核对 / 差异 / 待确认）
```

按 `地区 + 参保主体` 分组：

- 分组行：
  - 显示主体名称（粗体）+ 地区（灰字）
  - 支持展开 / 收起
  - 操作：详情 / 对账
- 子行：
  - 显示规则名称、当前账单月份、各统计数字
  - 操作：详情

### 9.5 颜色与状态表达

子行颜色规则：

- 已对账且无差异：浅绿
- 已对账且有差异：浅黄
- 已归档：灰色

金额颜色规则：

- 已匹配：绿色
- 差异：红色
- 待确认：橙色

### 9.6 页面行为要求

- 分组支持展开 / 收起
- 分组支持批量选择
- 支持分组一键对账
- 支持跳转单规则明细页，URL 必须携带当前账单月份 `month`
- 支持跳转主体级明细页，URL 必须携带当前账单月份 `month`

当前原型工具栏包含：

- `📥 导入台账`
- `▶ 开始对账`
- `✅ 归档结果`
- `📦 归档记录`
- `📤 导出`

功能说明：

## 11. 台账导入需求

### 11.1 导入流程

导入为两步流程：

**Step 1：选择参数 + 上传**

- 参保主体
- 参保规则
- 台账月份（自动继承当前账单月份，只读不可修改）
- 文件上传（拖拽或点击）

**Step 2：预览确认**

- 文件名
- 成功条数
- 错误条数
- 错误明细（行号 + 原因）
- 前 10 条预览
- 判重覆盖提示

### 11.2 导入格式要求

以主 spec 和导入专题为准，最小必填字段为：

- 姓名
- 身份证
- 险种
- 应缴月份
- 金额

可选字段：

- 单位金额
- 个人金额
- 备注

### 11.3 字段标准化

导入时必须进行：

1. 身份证标准化：
   - 去空格
   - 去首尾空白
   - 小写 `x` 转大写 `X`
2. 险种标准化：
   - 通过别名映射成标准险种
3. 应缴月份标准化：
   - 统一成 `YYYY-MM`
4. 金额标准化：
   - 四舍五入到分

### 11.4 判重与覆盖规则

判重维度：

- `参保主体 + 参保规则 + 账单月份`

规则：

- 同一维度只保留一份台账
- 重复导入时提示覆盖
- 覆盖后重新进入后续对账流程

### 11.5 导入异常

以下情况必须阻止导入：

- 必填字段缺失
- 身份证格式不正确
- 险种无法标准化
- 应缴月份格式错误
- 金额不是有效数字

以下情况可导入但要预警：

- 身份证相同但姓名不同
- 金额为 0
- 同一文件内相同主键重复

## 12. 匹配规则

### 12.1 主匹配口径

正式匹配输入口径为：

- `身份证 + 险种 + 应缴月份`

金额规则：

- 按总金额精确匹配
- 一分不差

补充口径：

- 自动匹配只在当前 `参保主体 + 参保规则 + 账单月份` 上下文内执行
- 匹配顺序固定为：标准化 → 主分组 → 金额桶处理 → 回填 → 统计
- 系统不做姓名模糊匹配、不做接近金额自动配对、不跨主体/规则/月份混配
- 对补缴、调基补差这类 `payableMonth` 初始为空的系统记录，本轮分组时按当前页面账单月份参与匹配；仅匹配成功后才正式回填应缴月份
- 同一成员、同一险种、当前账单月份、同一费款所属期下，系统侧与台账侧金额合计相等时，只允许在统一“确认核对”抽屉中人工确认，不自动 MATCHED；形态包含 1:1、一对多、多对一、多对多

### 12.2 自动匹配规则

1. 唯一 1:1 金额一致
   - 系统侧费用类型为汇缴，且未被 `forcePending` 标记：自动进入 MATCHED
   - 系统侧费用类型为补缴 / 调基补差：进入 PENDING
   - `forcePending=true`：进入 PENDING

2. 多笔同金额
   - 进入 PENDING
   - 需人工确认配对

3. 多笔合计金额相等
   - 不由系统自动生成组合组
   - 需用户在统一“确认核对”抽屉中选择系统侧和台账侧明细，可形成一对多、多对一或多对多
   - 确认时校验同成员、同险种、当前账单月份、同费款所属期、合计金额相等

4. 系统有台账无
   - 汇缴：DIFF(system_more)
   - 补缴 / 调基补差：UNMATCHED

5. 台账有系统无
   - DIFF(ledger_more)

6. 同分组存在金额不一致
   - DIFF(amount_mismatch)

补充说明：

- `amount_mismatch` 的定义是“先处理所有唯一 1:1 金额桶后，剩余仍无法闭合的残差进入差异；其中汇缴唯一 1:1 自动 MATCHED，补缴 / 调基补差 / forcePending 进入 PENDING”
- 同一主分组内允许同时存在 `MATCHED`、`PENDING`、`DIFF`
- 同金额多笔场景必须进入人工确认，不允许系统直接按列表顺序落正式结果
- `forcePending` 记录即使金额满足唯一 1:1，也不自动进入 `MATCHED`

### 12.3 无台账也可开始对账

即使未导入台账，也允许点击开始对账：

- 汇缴记录进入 DIFF(system_more)
- 补缴 / 调基补差进入 UNMATCHED

### 12.4 forcePending 机制

用于演示或需要人工审核的场景：

- 即使金额可自动匹配，也强制进入 PENDING

该机制在研发实现时可作为“需人工审核”能力的预留表达。

### 12.5 匹配结果回填

汇缴唯一 1:1 自动匹配成功后：

- 系统侧和台账侧进入 `MATCHED`
- 写入正式匹配关系
- 回填 `payableMonth / payableMonthInferred`

自动识别待确认后：

- 系统侧和台账侧进入 `PENDING`
- 不写入正式匹配关系
- 不正式回填 `payableMonth / payableMonthInferred`

用户确认核对成功后：

- 系统侧需回填：
  - `payableMonth`
  - `matchedLedgerId`
  - `matchStatus`
- 台账侧需回填：
  - `feeTypeInferred`
  - `payableMonthInferred`
  - `matchedSystemId`
  - `matchStatus`

强制核对后：

- 仅系统侧转为 `MATCHED`
- 台账侧不自动生成匹配成功关系
- 系统侧需保存强制前原始状态，用于取消恢复

## 13. 状态机与状态约束

### 13.1 状态定义

| 状态 | 含义 |
|------|------|
| UNMATCHED | 未参与匹配或暂未找到可确认台账 |
| MATCHED | 已核对成功 |
| PENDING | 待人工确认 |
| DIFF | 存在差异 |
| PAID | 已付款 |

### 13.2 状态流转

```text
UNMATCHED → MATCHED / PENDING / DIFF
PENDING → MATCHED
DIFF → MATCHED（仅系统侧强制核对）
MATCHED → 核对前原始状态（取消核对，仅未归档）
MATCHED + archived=true → 归档结果
MATCHED(已归档) → PAID
```

### 13.3 关键约束

- `archived` 是独立布尔标记，不是 `matchStatus`
- 已归档记录不可取消核对
- 已付款记录不可取消核对
- 台账侧无强制核对能力
- PAID 记录保留归档批次信息

## 14. 配对确认、差异处理与强制核对

### 14.1 配对确认抽屉

触发条件：

- `PENDING / UNMATCHED / DIFF` 记录点击 `确认核对`
- 系统侧或台账侧勾选一笔或多笔记录后点击 `批量确认核对`

抽屉内容：

- 基本信息区
- 系统侧明细区
- 台账侧明细区
- 合计校验区
- 底部确认按钮

关键规则：

- 用户选择系统侧和台账侧明细形成一次确认核对关系，覆盖 1:1、一对多、多对一、多对多
- 确认前必须校验：同成员、同险种、当前账单月份、同费款所属期，且系统金额合计 = 台账金额合计
- 确认成功后，参与记录全部转 MATCHED，并绑定同一个 `matchGroupId`
- 取消任一匹配组内记录时，同一 `matchGroupId` 下的系统侧和台账侧记录一起恢复至核对前原始状态

### 14.2 差异详情抽屉

触发条件：

- DIFF 记录点击 `详情`

内容包括：

- 基本信息
- 金额对比
- 差异分析
- 候选台账选择器

### 14.3 强制核对

仅系统侧支持，且仅作为批量操作栏兜底能力，不在系统侧单行操作列展示：

- DIFF 记录
- UNMATCHED 记录

规则：

- 操作员在系统侧批量操作栏选择一笔或多笔 `DIFF / UNMATCHED` 明细后，打开确认弹窗并确认执行
- 执行后：
  - `matchStatus = MATCHED`
  - `forceMatched = true`
  - `payableMonth` 回填为当前对账账单月份
- 取消强制核对时恢复至核对前原始状态

批量操作仅对当前 Tab 可见记录生效。

### 15.1 按钮集合

- 全选
- 批量确认核对
- 批量取消核对
- 强制核对

### 15.2 生效规则

| 按钮 | 适用记录 |
|------|----------|
| 批量确认核对 | PENDING |
| 批量取消核对 | MATCHED（未归档） |
| 强制核对 | DIFF / UNMATCHED（仅系统侧） |

- 批量强制核对时，自动忽略不符合条件的记录
- 按钮是否可点击取决于当前选中记录状态
- 强制核对仅保留在系统侧批量操作栏，允许选择一笔或多笔强制核对

## 16. 归档与付款

### 16.1 归档规则

归档对象：

- `MATCHED`
- `archived = false`

归档结果：

- 创建新归档批次
- 标记记录 `archived = true`
- 写入 `archiveBatchId`

### 16.2 归档确认

当页面仍存在未处理的 PENDING 或 DIFF 记录时：

- 弹出确认框
- 允许继续归档
- 但只归档已核对记录

### 16.3 归档批次抽屉

抽屉需展示：

- 批次名称
- 账单月份
- 记录数
- 金额
- 归档时间
- 操作：
  - 查看此批次
  - 申请付款

### 16.4 付款规则

付款从归档批次发起：

- 批次内 `MATCHED + archived=true` 的记录变更为 `PAID`
- 已付款记录不可再次付款
- 已付款保留归档批次信息

## 17. 异常与边界场景

### 17.1 导入相关

| 场景 | 预期处理 |
|------|---------|
| 未选台账月份上传 | 提示“请先选择台账月份” |
| 文件格式不支持 | 拒绝导入 |
| 空文件 | 提示无有效数据 |
| 必填字段缺失 | 标记错误行 |
| 金额格式错误 | 标记错误行 |
| 应缴月份格式错误 | 标记错误行 |

### 17.2 匹配相关

| 场景 | 预期处理 |
|------|---------|
| 同金额多笔 | 进入 PENDING |
| 汇缴唯一 1:1 金额一致 | 自动进入 MATCHED |
| 补缴 / 调基补差唯一 1:1 金额一致 | 进入 PENDING，点击「确认核对」后才转 MATCHED |
| 同成员同险种同账单月份同费款所属期，系统侧与台账侧合计金额相等 | 允许在统一确认抽屉中人工确认，覆盖一对多、多对一、多对多 |
| 补缴 / 调基补差无台账 | UNMATCHED |
| 汇缴无台账 | DIFF(system_more) |
| 台账无系统记录 | DIFF(ledger_more) |
| 同组部分金额一致、部分不一致 | 先处理可自动核对或待确认候选，剩余进入 DIFF(amount_mismatch) |
| 标记 forcePending 的唯一金额记录 | 与普通唯一 1:1 一样进入 PENDING，不直接 MATCHED |
| 同一系统记录重复配对 | 阻止并提示冲突 |

### 17.3 核对与归档相关

| 场景 | 预期处理 |
|------|---------|
| 已归档记录取消核对 | 拦截 |
| 已付款记录取消核对 | 不提供入口 |
| 无待归档记录 | Toast 提示 |
| 已全部付款的批次再次付款 | 提示无可付款记录 |

### 17.4 大数据场景

| 场景 | 预期处理 |
|------|---------|
| 数据量较大（>1000 条） | 显示进度反馈，分批处理 |
