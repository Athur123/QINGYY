# 社保对账复核 PRD

## Problem Statement

青阳云HRO系统根据员工参保异动自动生成应缴费用（汇缴/补缴/调基补差），但补缴和补差类型缺少应缴月份信息。社保局台账（Excel导入）记录了应缴月份，但不区分费用类型。社保专员需要将系统侧费用与台账侧账单逐笔核对，确保缴费金额一致，并将核对结果归档用于后续付款申请。

核心矛盾：**系统有费用类型、无应缴月份；台账有应缴月份、无费用类型**。双方共同拥有身份证号、险种、金额。同一员工同一险种可能产生多笔费用（1笔汇缴 + N笔补缴 + M笔调基补差），金额相同的记录无法自动区分类型。

当前流程依赖人工逐行比对 Excel，效率低、易出错，缺乏批量操作和归档追溯能力。

## Solution

提供两层结构的对账复核系统：

1. **汇总列表页**：按参保规则维度展示对账概览，支持批量导入台账、批量对账、批量归档
2. **明细核对页**：系统侧/台账侧 Tab 分离展示，按 `身份证+险种+应缴月份` 分组进行精确金额配对，支持人工确认配对、强制核对、差异处理

核对完成后将已匹配记录归档为批次，同一账单月份支持多次归档（应付增量费用场景），归档后可按批次筛选和追溯。支持从已归档批次发起付款申请，完成付款后将记录标记为已付款状态。

## User Stories

### 台账导入
1. 作为社保专员，我想导入社保局台账 Excel，以便与系统侧费用进行比对
2. 作为社保专员，我想在导入时选择参保主体、参保规则和台账月份，以便数据正确归属
3. 作为社保专员，导入前必须先选择台账月份，否则无法上传文件
4. 作为社保专员，我希望能看到导入预览（成功条数、错误条数、错误行详情），以便在确认前修正问题
5. 作为社保专员，我希望系统对身份证格式（去空格、X大写）、险种名称（别名映射为标准名）、金额格式（保留2位小数）自动标准化，减少手动处理
6. 作为社保专员，当我重复导入同一参保主体+规则+月份的台账时，希望系统提示是否覆盖已有数据
7. 作为社保专员，多次导入不同月份的台账时，数据应累加而非覆盖

### 对账匹配
8. 作为社保专员，我想点击"开始对账"后自动执行金额配对，系统侧和台账侧按 `身份证|险种|应缴月份` 分组，同金额精确匹配，双向回填信息（系统回填 payableMonth，台账回填 feeTypeInferred）
9. 作为社保专员，无需导入台账也可执行对账——系统侧费用基于员工异动已预计算，无台账时全部标记为差异或未匹配
10. 作为社保专员，当同一分组内金额有唯一匹配时，系统自动标记为"已核对"
11. 作为社保专员，当同一分组内金额有多笔同金额记录时，系统标记为"待确认"并提供智能推荐配对（按列表顺序推荐）
12. 作为社保专员，我可以在配对抽屉中手动调整系统侧和台账侧的映射关系（下拉选择台账记录），然后确认配对
13. 作为社保专员，当系统侧费用在台账中无匹配时，系统区分处理：
    - 汇缴费用 → 标记为"差异"（system_more），因汇缴应在当前月份有对应台账，需人工核查
    - 补缴/调基补差 → 标记为"未匹配"，因可能对应其他应缴月份的台账，属正常状态
14. 作为社保专员，当台账侧记录在系统中无对应时，系统标记为"差异"（ledger_more）
15. 作为社保专员，当系统侧与台账侧金额不一致时，系统标记为"差异"（amount_mismatch）并显示最近似金额的差额

### 强制核对
16. 作为社保专员，对于已确认为正确的差异或未匹配记录，我想"强制核对"直接将系统侧记录标记为已核对，跳过台账匹配要求
17. 作为社保专员，台账侧记录**不允许**强制核对——台账必须有对应的系统侧账单才能进行匹配核对
18. 作为社保专员，强制核对操作前需要确认对话框，展示将受影响的记录明细
19. 作为社保专员，强制核对后可以取消操作，恢复到原始的差异/未匹配状态和原始字段值
20. 作为社保专员，我可以在系统侧底部批量选中记录进行批量强制核对

### 明细核对
21. 作为社保专员，我想在明细页通过 Tab 切换查看系统侧账单和台账侧账单，两侧各自独立维护筛选状态
22. 作为社保专员，我想按核对状态（全部/已核对/待确认/未匹配/差异，各状态带金额统计）和费用类型（全部/汇缴/补缴/调基补差）筛选记录
23. 作为社保专员，已核对记录应在系统侧和台账侧同时展示，并带有跨引用标记（系统侧显示 ↔ Txxx，台账侧显示 ↔ Sxxx）
24. 作为社保专员，我想对系统侧待确认记录批量确认核对，对差异记录批量操作
25. 作为社保专员，我想点击差异记录的详情按钮打开右侧抽屉，查看金额对比、差异分析和可能原因

### 归档批次
26. 作为社保专员，我想点击"归档结果"将当前所有已核对且未归档的记录归档为一个新批次
27. 作为社保专员，同一账单月份可能有增量费用（新员工入保、补缴等），我希望能多次归档，每次自动生成新批次（第1批、第2批...）
28. 作为社保专员，归档操作时检查是否存在未处理的待确认或差异记录，若有则弹窗确认后仅归档已核对记录
29. 作为社保专员，我想通过工具栏"归档记录"按钮打开右侧抽屉，查看所有历史归档批次列表（批次号、账单月份、记录数、金额、归档时间）

31. 作为社保专员，我想在筛选栏通过"归档批次"下拉选择器（包含全部批次 + 各批次选项）过滤记录
32. 作为社保专员，已归档记录状态列显示"已归档"（灰色 badge），操作列从"取消"变为"详情"
33. 作为社保专员，已归档记录不可取消核对以防止误操作
34. 作为社保专员，无待归档记录时点击归档应提示"无待归档记录"
35. 作为社保专员，无批次时"归档记录"按钮为禁用状态

## 核对状态机

```
                    ┌──────────┐
                    │ UNMATCHED │  ← 初始状态
                    └─────┬────┘
                          │ 执行对账
              ┌───────────┼───────────┐
              │           │           │
        ┌─────┴────┐ ┌────┴─────┐ ┌──┴──────┐
        │  MATCHED  │ │ PENDING  │ │  DIFF   │
        │ (唯一配对) │ │ (待确认)  │ │ (差异)  │
        └─────┬────┘ └────┬─────┘ └────┬─────┘
              │           │           │
              │     确认核对│     强制核对│
              │           │           │
              └── 取消核对 ─┘           │
              │  ↑ (已归档不可取消)     │
              │           │           │
              ▼ 归档      ▼ 归档      ▼ 归档
        ┌──────────┐
        │ MATCHED   │  ← archived=true, archiveBatchId
        │ (已归档)   │
        └─────┬────┘
              │ 付款申请（后续版本）
              ▼
        ┌──────────┐
        │   PAID    │  ← matchStatus=PAID
        │ (已付款)   │
        └──────────┘
```

**状态转换规则**：
- UNMATCHED → MATCHED：唯一金额配对成功 / 人工确认配对 / 强制核对
- UNMATCHED → PENDING：同金额多笔
- UNMATCHED → DIFF：金额无匹配或金额不一致
- PENDING → MATCHED：用户在配对抽屉确认
- DIFF → MATCHED：系统侧强制核对
- MATCHED → PENDING：取消核对（仅未归档记录）
- MATCHED → archived=true：归档操作，matchStatus 保持 MATCHED
- MATCHED(archived=true) → PAID：从归档批次发起付款后，matchStatus 变更为 PAID，archived 和 archiveBatchId 保留

**关键约束**：
- `archived` 不是 matchStatus 的值，是独立布尔字段
- 已归档记录不可取消核对（`doCancelMatch` 拦截 + toast 提示）
- 已归档记录操作列显示"详情"而非"取消"
- 台账侧不允许强制核对

## 匹配算法

### 核心流程

```
Step 1: 数据标准化
  ├─ 身份证：去空格、X 大写、trim
  ├─ 险种：通过映射表标准化（别名 → 标准名称）
  └─ 金额：保留 2 位小数

Step 2: 分组
  ├─ key = idCard + "|" + insuranceType + "|" + billingMonth
  ├─ systemGroups: Map<key, SystemRecord[]>
  └─ ledgerGroups: Map<key, LedgerRecord[]>

Step 3: 组内匹配
  ├─ 统计每个金额的出现次数
  ├─ 按金额遍历：
  │   ├─ 系统 1:1 台账 → 自动 MATCHED（回填双方信息）
  │   ├─ 多笔同金额 → PENDING（进入配对抽屉）
  │   ├─ 系统有、台账无：
  │   │   ├─ 汇缴 → DIFF(system_more)
  │   │   └─ 补缴/调基补差 → UNMATCHED
  │   ├─ 台账有、系统无 → DIFF(ledger_more)
  │   └─ 金额不一致 → DIFF(amount_mismatch)，记录最近似金额差额

Step 4: forcePending 后期处理
  └─ forcePending=true 的记录从 diffs 中移除，重新标记为 PENDING
```

## 导入流程

```
Step 1: 选择导入参数
  ├─ 参保主体（下拉选择）
  ├─ 参保规则（下拉选择）
  └─ 台账月份（下拉选择，必选，未选时禁止上传）

Step 2: 上传 Excel
  ├─ 支持 .xlsx / .xls
  ├─ 拖拽或点击上传
  └─ 格式校验：
      ├─ 必填列检查（姓名、身份证、险种、应缴月份、金额）
      ├─ 身份证格式标准化
      ├─ 险种名称映射
      └─ 金额格式检查

Step 3: 判重检查
  └─ 参保主体 + 参保规则 + 应缴月份 是否已有台账
      ├─ 有 → 弹窗提示"该维度下已有台账数据，是否覆盖？"
      └─ 无 → 继续

Step 4: 导入预览
  ├─ 展示：成功条数、错误条数、错误行详情
  ├─ 按规则拆分的预览表格（规则名称、月份、导入记录数、总金额）
  └─ 用户确认后写入 ledgerRecords
```

## 交互规范

### Tab 切换

- 系统侧账单 / 台账侧账单两个 Tab
- `.bill-tabs` 使用 `position: sticky; top: 0; z-index: 100` 吸顶
- 各自独立维护核对状态筛选和费用类型筛选
- Tab 计数实时更新

### 筛选器

- **核对状态筛选**：全部 / 已核对 / 待确认 / 未匹配 / 差异，每个选项附带金额统计
- **费用类型筛选**：全部 / 汇缴 / 补缴 / 调基补差
- **归档批次筛选**：下拉选择器，选项随归档操作动态更新，默认"全部批次"

### 表格行操作

| 状态 | 操作按钮 |
|------|---------|
| UNMATCHED | 强制核对（仅系统侧）、详情（台账侧） |
| PENDING | 确认核对 |
| MATCHED（未归档） | 取消 |
| MATCHED（已归档） | 详情 |
| DIFF（系统侧） | 强制核对 |
| DIFF（台账侧） | 详情 |

### 抽屉组件

| 抽屉 | 触发方式 | 内容 |
|------|---------|------|
| 配对确认抽屉 | PENDING 行点击"确认核对" | 智能推荐配对 + 手动调整映射 + 确认按钮 |
| 差异详情抽屉 | DIFF 行点击"详情" | 金额对比、差异分析、可能原因、确认核对按钮 |
| 归档批次抽屉 | 工具栏"归档记录"按钮 | 批次列表 + "查看此批次"按钮（关闭抽屉并筛选） |

### 弹窗组件

| 弹窗 | 触发条件 | 内容 |
|------|---------|------|
| 归档确认 | 存在 PENDING/DIFF 时点击归档 | 提示条数，确认后仅归档 MATCHED 记录 |
| 强制核对确认 | 点击强制核对 | 展示待操作记录明细（姓名、身份证、险种、金额） |
| 导入覆盖确认 | 判重命中 | 提示已有数据，确认覆盖 |

## Edge Cases

| 场景 | 处理方式 |
|------|---------|
| 同一台账重复导入 | 参保主体+参保规则+应缴月份维度判重，提示覆盖 |
| 核对成功后取消 | 系统侧清空 payableMonth，台账侧清空 feeTypeInferred，恢复 PENDING |
| 身份证格式不一致 | 导入时标准化（去空格、X大写） |
| 险种名称不一致 | 映射表标准化，无法识别标记"未知" |
| 台账导入空文件 | 提示"文件中无有效数据" |
| 必填字段为空 | 标记为格式错误行，跳过 |
| 已归档记录取消核对 | `doCancelMatch()` 拦截，toast "已归档记录不可取消核对" |
| 无待归档记录 | toast "无待归档记录"，不创建批次 |
| 强制核对后取消 | 恢复 `_originalMatchStatus`、`_originalDiffType`、`_originalDiffAmount` |
| 台账侧强制核对 | 按钮不存在，台账侧必须有对应系统侧记录 |
| 补缴/调基补差无台账 | 标记 UNMATCHED 而非 DIFF，可在后续月份继续匹配 |
| 只有 PENDING/DIFF 无 MATCHED | toast "无已核对记录可归档" |

## Error Handling

| 错误场景 | 处理方式 |
|----------|---------|
| Excel 格式不支持 | 拒绝导入，提示"请使用 .xlsx 或 .xls 文件" |
| 身份证校验失败 | 标记错误行，跳过，提示行号和原因 |
| 金额格式错误 | 标记错误行，跳过 |
| 应缴月份格式错误 | 标记错误行，跳过，提示使用 YYYY-MM 格式 |
| 未选台账月份就上传 | 提示"请先选择台账月份" |
| 匹配执行异常 | 提示"对账过程发生错误"，回滚核对状态 |

## Testing Strategy

### 测试范围

1. **功能测试**：Playwright 浏览器自动化，验证完整用户流程
2. **状态机测试**：覆盖所有状态转换路径和边界条件
3. **UI 验证**：CSS 无 404、页面渲染正确、响应式断点
4. **数据一致性**：归档前后记录数、批次金额聚合、取消后还原

### 关键测试场景

**台账导入**：
- 正常导入含 17 条记录的 Excel
- 空文件导入 → 错误提示
- 缺少必填列 → 格式错误行跳过
- 判重覆盖 → 确认弹窗
- 多次导入不同月份 → 数据累加

**对账匹配**：
- 全部 1:1 金额一致 → 全部自动 MATCHED
- forcePending 标记的记录 → 应进入 PENDING 而非自动 MATCHED
- 无台账数据 → 汇缴 DT DIFF、补缴/调基补差 UNMATCHED
- 台账独有记录 → DIFF(ledger_more)

**强制核对**：
- 系统侧 DIFF 行点击强制核对 → 确认弹窗 → 变为 MATCHED
- 取消强制核对 → 恢复 DIFF 状态
- 台账侧 DIFF 行 → 无强制核对按钮
- 批量强制核对 → 仅 DIFF/UNMATCHED 行参与

**归档批次**：
- 无差异时归档 → 创建第1批
- 有 DIFF/PENDING 时归档 → 确认弹窗后仅归档 MATCHED
- 新增记录后再次归档 → 创建第2批
- 空归档 → toast "无待归档记录"
- 归档记录取消核对 → 被拦截
- 批次筛选器 → 选择批次后表格正确过滤

**汇总列表**：
- 展开/收起分组 → 子行正确显示
- 批量勾选 → 批量对账/归档按钮状态正确
- 已归档规则 → 灰色行 + checkbox 禁用
