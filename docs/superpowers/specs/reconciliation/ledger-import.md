# 社保台账导入规范

> 状态说明：
> 这份文档是对账复核模块的导入/输入专题规范，主要用于定义台账文件格式、字段要求、校验规则和客户整理要求。
> 当前对账复核模块的完整主规格以 `type-month-matching.md` 为准；若导入字段口径、匹配规则或页面交互与主规格不一致，应优先采用 `type-month-matching.md`，并将本文档视为需要同步更新的补充文档。

## 一、适用范围

本规范服务于对账复核模块的台账导入环节，目标是让外部台账文件满足主 spec 的匹配输入要求。

在主 spec 下：

- 用户先在页面选择 `参保主体 + 参保规则 + 账单月份`
- 再上传台账文件
- 系统对文件做格式校验、字段标准化、批次判重和导入预览
- 导入成功后，台账记录进入后续对账流程

说明：

- `参保主体`、`参保规则`、`账单月份` 来自页面上下文，不要求文件中提供
- 台账文件本身只需要提供对账所需的明细字段
- 费用类型不从文件直接给出，后续由系统在匹配过程中回填

## 二、文件规格

### 2.1 格式要求

| 项目 | 要求 |
|-----|-----|
| 文件格式 | Excel (`.xlsx` / `.xls`) 或 CSV |
| 编码 | CSV 使用 UTF-8 |
| 单文件上限 | 10 万行 |
| 导入范围 | 单次导入对应当前所选的 `参保主体 + 参保规则 + 账单月份` |
| 覆盖规则 | 同一 `参保主体 + 参保规则 + 账单月份` 维度只保留一份台账，重复导入时提示覆盖 |

### 2.2 必填字段

主 spec 要求的最小必填列如下：

| 序号 | 字段名 | 必填 | 格式 | 说明 |
|:---:|--------|:---:|------|------|
| 1 | 姓名 | ✓ | 文本，不超过 50 字符 | 员工姓名 |
| 2 | 身份证 | ✓ | 文本 | 18 位身份证号，导入时会做标准化 |
| 3 | 险种 | ✓ | 文本 | 险种名称，支持别名映射 |
| 4 | 应缴月份 | ✓ | `YYYY-MM` | 作为主匹配月份字段 |
| 5 | 金额 | ✓ | 数值，保留 2 位小数 | 账单明细总金额 |

### 2.3 可选字段

| 字段名 | 格式 | 说明 |
|--------|------|------|
| 单位金额 | 数值，保留 2 位小数 | 用于辅助展示或核查 |
| 个人金额 | 数值，保留 2 位小数 | 用于辅助展示或核查 |
| 备注 | 文本 | 导入后保留为备注信息 |

### 2.4 列名兼容映射

为兼容不同地区社保导出格式，系统允许将常见原始列名映射到主 spec 标准字段：

| 标准字段 | 兼容列名示例 |
|----------|-------------|
| 姓名 | 姓名 |
| 身份证 | 身份证 / 身份证号 / 证件号 / 证件号码 |
| 险种 | 险种 / 险种名称 |
| 应缴月份 | 应缴月份 / 费用月份 / 费款所属期 / 申报月份 / 汇缴月份 |
| 金额 | 金额 / 合计金额 / 应缴金额 / 实缴金额 |
| 单位金额 | 单位金额 / 单位缴费额 / 单位缴费金额 / 单位实缴金额 / 单位月缴额 |
| 个人金额 | 个人金额 / 个人缴费额 / 个人缴费金额 / 个人实缴金额 / 个人月缴额 |

如果原始文件没有 `金额` 列，但提供了 `单位金额` 和 `个人金额`，应在整理模板阶段合并出 `金额` 列，再按标准模板导入。

## 三、险种标准化口径

### 3.1 标准险种名称

导入后的标准险种名称应与主 spec 保持一致：

- 养老
- 医疗
- 失业
- 工伤
- 生育
- 大额医疗
- 公积金

## 四、标准模板

### 4.1 标准长表格式（推荐）

每行 = 一条台账明细（一个员工 × 一个险种 × 一个应缴月份）

| 姓名 | 身份证 | 险种 | 应缴月份 | 金额 | 单位金额 | 个人金额 | 备注 |
|------|--------|------|---------|------|---------|---------|------|
| 张三 | 430105199001011234 | 养老 | 2026-04 | 635.25 | 423.50 | 211.75 | 汇缴 |
| 张三 | 430105199001011234 | 医疗 | 2026-04 | 280.00 | 180.00 | 100.00 | 汇缴 |
| 李四 | 430106199102022345 | 养老 | 2026-01 | 334.00 | 234.00 | 100.00 | 补缴 |
| 王五 | 430107199203033456 | 公积金 | 2026-04 | 600.00 | 300.00 | 300.00 | 正常 |

### 4.2 外部导出格式兼容说明

部分城市社保局导出台账不是标准模板，但只要能映射到主 spec 的核心字段，仍可整理后导入。

常见情况：

- 原文件使用 `费用月份 / 费款所属期 / 申报月份 / 汇缴月份`，统一映射到 `应缴月份`
- 原文件使用 `养老保险 / 医疗保险 / 住房公积金` 等长名称，统一映射到标准险种
- 原文件拆成 `单位金额 + 个人金额`，整理模板时汇总为 `金额`
- 原文件保留了更多字段，可按需保留在备注列或作为人工辅助信息，不参与主匹配

## 五、导入校验与标准化

### 5.1 阻断型校验

以下情况应阻止导入：

| 校验项 | 错误提示 | 处理方式 |
|--------|---------|---------|
| 必填字段为空 | 第 N 行：[字段名] 为必填项，不能为空 | 修正后重新导入 |
| 身份证格式错误 | 第 N 行：身份证格式不正确 | 修正后重新导入 |
| 险种无法标准化 | 第 N 行：险种 `XXX` 无法识别 | 修正为支持的险种名称 |
| 应缴月份格式错误 | 第 N 行：应缴月份格式不正确，应为 `YYYY-MM` | 修正后重新导入 |
| 金额不是有效数字 | 第 N 行：金额格式不正确 | 修正后重新导入 |

### 5.2 非阻断预警

以下情况可导入，但应提示人工关注：

| 预警类型 | 提示信息 | 处理建议 |
|---------|---------|---------|
| 身份证相同但姓名不同 | 第 N 行与第 M 行身份证相同但姓名不同，请核实 | 人工确认是否同一人 |
| 金额为 0 | 第 N 行金额为 0，请确认是否正常 | 人工确认 |
| 同一文件内同主键重复 | 同一 `身份证 + 险种 + 应缴月份` 出现多条记录 | 检查是否重复导出或应合并 |

### 5.3 标准化规则

导入时应执行与主 spec 一致的标准化：

1. 身份证：去空格、去前后空白、`x` 转大写 `X`
2. 险种：通过别名映射转换为标准险种名称
3. 应缴月份：统一格式为 `YYYY-MM`
4. 金额：四舍五入到分，保留 2 位小数

补充说明：

- 导入完成后的台账“应缴月份”会作为后续匹配阶段的主分组月份字段
- 若应缴月份标准化失败，该行不能进入自动匹配
- 若险种标准化失败，该行不能进入自动匹配
- 若金额标准化失败，该行不能进入自动匹配

## 六、导入后进入主 spec 的方式

### 6.1 导入后台账记录的角色

导入成功后的台账记录会进入主 spec 定义的临时对账表，并具备以下特征：

- 初始状态为 `UNMATCHED`
- 不直接携带费用类型
- 后续在对账过程中回填费用类型和其他推断字段
- 导入批次、导入人、参保主体、参保规则等元数据来自页面上下文

### 6.2 与主 spec 一致的匹配输入

导入后的正式匹配输入口径为：

- 匹配主键：`身份证 + 险种 + 应缴月份`
- 金额比对：按总金额精确匹配，一分不差
- 匹配结果：进入主 spec 的 `MATCHED / PENDING / DIFF / PAID` 状态流转

导入质量对匹配结果的直接影响如下：

| 输入问题 | 对匹配的影响 | 处理要求 |
|----------|-------------|---------|
| 身份证未标准化 | 无法进入正确主分组，可能产生假性 `ledger_more` | 导入阶段阻断或修正 |
| 险种别名未识别 | 无法与系统侧进入同一主分组 | 导入阶段阻断 |
| 应缴月份格式错误 | 无法进入正确月份分组，可能错配到错误账单月份 | 导入阶段阻断 |
| 金额小数位异常 | 可能导致本应匹配的记录落入不同金额桶 | 导入阶段统一四舍五入到分 |
| 同一文件内重复明细 | 可能造成一对多 / 多对多假性 `PENDING` | 导入时预警，必要时由客户整理去重 |

补充边界：

- 同一 `身份证 + 险种 + 应缴月份` 出现多条台账记录并不一定是错误，可能是真实多笔；系统不会在导入阶段自动合并
- 是否进入 `MATCHED`、`PENDING` 或 `DIFF`，仍以主 spec 的金额桶决策为准
- “最接近金额”不在导入阶段处理，导入阶段只负责把字段整理成可进入匹配的标准输入

说明：

- 本文档不单独定义新的匹配算法
- 正式匹配、待确认、强制核对、归档等规则均以 `type-month-matching.md` 为准

### 6.3 覆盖与判重

与主 spec 保持一致：

- 同一 `参保主体 + 参保规则 + 账单月份` 维度只保留一份台账
- 重复导入时应提示是否覆盖原台账
- 覆盖后重新进入后续对账流程

## 七、客户整理要求说明

### 7.1 发送给客户的要求模板

```text
【社保台账整理要求】

请按以下要求整理并提供贵司指定账单月份的社保公积金台账数据：

一、文件要求
1. 格式：Excel（.xlsx / .xls）或 CSV
2. 命名建议：公司名称_账单月份_台账

二、必填字段
1. 姓名
2. 身份证
3. 险种
4. 应缴月份（YYYY-MM）
5. 金额

三、可选字段
1. 单位金额
2. 个人金额
3. 备注

四、整理要求
1. 一个员工同一险种同一应缴月份，对应一条台账明细
2. 身份证请提供完整值，不要带空格
3. 险种名称可保持原始导出值，系统会按支持的别名映射标准化
4. 如果原始文件没有金额总列，请先将单位金额和个人金额汇总为金额列
5. 如原始导出格式无法直接整理，请附上原始文件
```

### 7.2 常见整理问题及处理建议

| 客户问题 | 处理建议 |
|---------|---------|
| 原文件只有单位金额和个人金额，没有总金额 | 先整理出 `金额 = 单位金额 + 个人金额` 后再导入 |
| 原文件月份列名不是“应缴月份” | 按模板整理为标准列，或在导入映射中确认对应关系 |
| 原文件险种名称较长 | 保留原值即可，由系统按别名映射标准化 |
| 原文件有多个 Sheet | 可先合并为标准长表，再导入 |
| 只能导出 PDF | 先转换为可编辑表格，再按模板整理 |
