# 员工档案”动态记录”模块 PRD

## 1. 业务背景与目标
### 1.1 背景
在企业的人力资源与薪税管理中，员工的档案数据贯穿其整个在职生命周期。为了满足企业内部的审计合规要求，降低跨部门（如 HR 与财务）协作时的沟通成本，并确保数据的可追溯性，我们需要在“员工档案详情页”中引入一个全局的“动态记录”模块。

### 1.2 核心目标
* **全局溯源**：集中呈现员工在所有模块（基础信息、组织架构、薪税档案、参保信息等）下的状态变更与修改日志。
* **精细化 Diff**：精确到“字段级”的变更追踪，清晰展示“谁（Who）、在什么时间（When）、通过什么渠道（How）、将什么字段从 A 改成了 B（What）”。
* **历史快照**：针对员工入职、转正等关键里程碑事件，提供当时全量数据的只读快照能力，防止因后续修改导致历史原始面貌丢失。

## 2. 核心业务规则与数据范围
### 2.1 数据涵盖范围
* 该时间线需涵盖该员工在系统内**所有业务模块**的变更记录（包含且不限于：基础信息、联系方式、工作信息、家庭成员、教育经历、工作经历、银行卡信息、合同协议、员工材料、薪税档案、参保档案等）。

### 2.2 记录与排序规则
* **倒序排列**：时间线必须严格按照操作时间（`OperateTime`）的**倒序**排列，即最新发生的操作展示在列表最上方。
* **数据聚合与子记录（Sub-record）**：
  * 当用户在同一时间点、同一次提交动作中，修改了多个模块或多个子层级的数据（如：批量修改了两个不同分公司的薪税档案主体），在时间轴上应记为**一次操作事件**。
  * 该事件内部通过多张“子记录卡片（Sub-record Card）”进行物理隔离和展示。

### 2.3 操作人与渠道规则
* **实名记录**：若为业务人员手动修改，需记录并展示人员姓名及角色（如：`张HR (管理员)`）。
* **系统触发**：若为系统定时任务、接口同步或自动化流转产生的变更（如：薪税系统接口回传申报成功状态），操作人需统一显示为 `系统自动触发`，并标明具体的渠道来源（如：`渠道: 薪税系统接口`）。

### 2.4 快照留存规则
* **重大节点全量快照**：仅针对“员工入职”、“员工转正”、“部门调动”、“员工离职”等生命周期内的重大节点，系统在触发时需生成一份当时的**全量静态数据快照**。
* **常规修改增量记录**：对于日常的普通编辑（如修改手机号、更新银行卡），系统**不生成**全量快照，仅记录发生变更的字段（Diff 增量），以优化存储空间。

## 3. UI/UX 布局与交互规范
### 3.1 页面布局与导航
* **定位**：作为员工详情页的顶级选项卡（Tab）之一，命名为“动态记录”。
* **整体结构**：采用垂直时间轴（Vertical Timeline）结合嵌套卡片（Card-based）的设计。
  * 顶部固定为条件筛选区。
  * 主体区域为向下延伸的动态事件列表。
  * 底部提供“加载更多”按钮或滚动分页机制。

### 3.2 筛选区交互
提供三个并排的筛选控件：
1. **时间范围筛选**：使用只读输入框触发，集成日历范围选择组件（如 Flatpickr）。输入框内需有明显的日历 Icon 提示。选中后按 `YYYY-MM-DD` 至 `YYYY-MM-DD` 格式回显。
2. **档案模块筛选**：下拉单选框。选项包含：全部模块（默认）、基础信息、联系方式、工作信息、家庭成员、教育经历、工作经历、银行卡信息、合同协议、员工材料、薪税档案、参保档案等（需与顶部业务 Tab 严格对齐）。
3. **操作人筛选**：下拉单选框。支持按具体人员名称或“系统自动触发”进行过滤。

### 3.3 时间轴列表与卡片设计
* **事件头（Header）**：
  * **时间轴圆点（Timeline Dot）**：使用不同颜色表达操作定性（蓝色-编辑/导入，绿色-新增/转正，橙色-调动/警告，红色-删除）。
  * **元数据**：单行展示“事件名称”、“操作人”、“操作渠道”、“操作时间”，中间以圆点分隔。
* **子记录卡片（Sub-record Card）**：
  * **卡片头部**：展示带背景色的动作 Tag（如 <span style="background:#EFF6FF;color:#2563EB">编辑</span>） + 具体的被修改对象名称（如 `薪税档案 (深圳分公司)`）。
  * **快照入口**：若该事件存在全量历史快照，头部右侧需提供“查看快照”的文字按钮。
* **字段变更明细（Change List）**：
  * **双列布局**：采用 CSS Grid 实现一行两列排布（移动端屏幕小于 768px 时响应式折叠为单列）。
  * **Diff 视觉强化**：
    * `字段名`：常规展示。
    * `旧值`：置灰文字 + 删除线。
    * `箭头`：SVG 右向箭头。
    * `新值`：浅绿色背景块 + 绿色加粗文字。
  * **特殊字体**：涉及长串数字（身份证、银行卡号等）必须使用等宽字体（Monospace）渲染。

### 3.4 历史快照抽屉交互
* 点击“查看快照”按钮后，从页面右侧滑出抽屉层（Drawer）。
* 抽屉滑出时，主页面需铺设半透明遮罩，阻止背景滚动与交互。
* 抽屉内以只读模式（剥离所有编辑按钮和表单输入框）完整渲染该历史节点的全貌数据。
* 支持点击右上角关闭按钮、点击遮罩层、或按 `ESC` 键关闭抽屉。

## 4. 数据结构与接口定义建议
### 4.1 核心数据结构 (Timeline Item)
为了支撑前端卡片式渲染与 Diff 展示，建议后端接口按以下聚合结构返回单条时间轴数据：

```json
{
  "eventId": "EVT_10029381",
  "eventName": "修改人员档案",        // 主事件名称
  "operateTime": "2026-04-23 10:05:00",
  "operatorName": "张HR",            // 操作人名称，若为系统则传"系统自动触发"
  "operatorRole": "管理员",          // 操作人角色
  "channel": "Web端",                // 操作渠道（Web端、薪税系统接口、Excel导入等）
  "snapshotId": "SNAP_88392",        // 快照ID（若无快照则为 null）
  "eventColor": "blue",              // 节点颜色（blue/green/orange/red）
  
  // 子记录卡片数组（同一次操作可能修改多个不同模块/实体）
  "subRecords": [
    {
      "actionType": "edit",          // 动作类型：add/edit/delete
      "actionLabel": "编辑",          // Tag 展示文案
      "targetName": "薪税档案 (深圳分公司)", // 被修改对象名称
      
      // 字段变更明细数组
      "changes": [
        {
          "fieldLabel": "申报状态",
          "oldValue": "未报送",      // 若为新增，oldValue 为 null
          "newValue": "申报中",      // 若为删除，newValue 为 null
          "isMono": false           // 是否需要等宽字体渲染（如身份证号）
        }
      ]
    }
  ]
}
```

### 4.2 接口交互建议
1. **列表查询接口**：采用基于游标（Cursor-based）或页码（Page-based）的分页查询，入参需包含员工 ID、日期区间（`startDate`, `endDate`）、模块分类（`moduleCode`）及操作人（`operatorId`）。
2. **快照获取接口**：前端点击“查看快照”时，通过 `snapshotId` 单独调用接口拉取历史全量结构化数据进行渲染。

## 5. 异常流与边界条件
### 5.1 数据为空 (Empty State)
* **无任何记录**：员工刚建档，尚未产生历史操作时，展示“暂无动态记录”的空状态提示及插画。
* **筛选无结果**：用户使用条件过滤后未匹配到数据时，展示“暂无符合条件的动态记录”。

### 5.2 极端数据边界
* **超长文本修改**：如果修改的字段是一个长文本（如“备注”），前端 `change-list` 需限制最大高度，并使用 `text-overflow: ellipsis` 截断，支持 Hover 或点击查看完整对比。
* **批量大规模修改**：如果一次 Excel 导入同时修改了该员工 50 个字段，前端不应截断，应允许该 `timeline-item` 在页面内自然拉长，由浏览器原生滚动条处理。

### 5.3 快照缺失与数据断层
* **快照过期或清理**：若历史快照由于数据清理策略被物理删除，点击“查看快照”时应给予 Toast 提示：“该历史快照已归档或清理，无法查看”。
* **操作人离职**：如果历史操作人已从系统中离职被删，操作人名称应回显为其离职前的历史快照名字（而非 ID 或空值）。
