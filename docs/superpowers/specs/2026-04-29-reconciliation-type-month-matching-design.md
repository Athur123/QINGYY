# 社保对账复核：类型/月份互推匹配设计

## Context

青阳云系统根据员工参保异动生成应缴费用，包含汇缴、补缴、调基补差三种类型。系统知道费用类型，但补缴和补差不知道应缴月份。税务台账（Excel导入）知道应缴月份，但不知道费用类型。

核心矛盾：系统有类型无月份，台账有月份无类型。双方共同拥有：身份证、险种、金额。

同一应缴月份内，一个员工同一险种可能产生多笔费用：1笔汇缴 + N笔补缴 + M笔补差。台账中同样有多笔，但无法区分类型。

金额特征（零头、特殊值）无法作为类型判断依据。同一金额在系统侧和台账侧可能各有多笔，无法确定对应关系。

## 约束确认

| 项目 | 确认值 |
|------|--------|
| 匹配范围 | 身份证 + 险种 + 应缴月份 |
| 匹配方式 | 精确金额匹配（一分不差） |
| 触发方式 | 台账导入后，用户手动点击"开始对账" |
| 匹配成功后 | 系统回填应缴月份，台账回填费用类型 |
| 同金额多笔 | 标记"待确认"，人工判断 |
| 待确认交互 | 智能推荐 + 批量映射 + 确认 |
| 对账输出 | 匹配成功归档 + 差异处理流程 |
| 判重规则 | 参保主体 + 参保规则 + 应缴月份 维度只保留一份台账 |
| 差异类型 | 系统多（台账无）、台账多（系统无）、金额不一致（双方都有但金额不同） |

### 核对状态机

```
              ┌──────────┐
              │ UNMATCHED │  ← 初始状态（导入后/取消匹配后）
              └─────┬────┘
                    │ 执行对账
              ┌─────┴─────────────────────┐
              │                           │
        ┌─────┴────┐               ┌──────┴─────┐
        │  MATCHED  │               │  PENDING   │  ← 同金额多笔
        │ (唯一配对) │               │ (待确认)    │
        └─────┬────┘               └──────┬─────┘
              │                           │ 用户确认核对
              │                    ┌──────┴─────┐
              │                    │  MATCHED   │
              │                    └──────┬─────┘
              │                           │
        ┌─────┴───────────────────────────┴─────┐
        │              ARCHIVED                  │
        └───────────────────────────────────────┘

  差异路径（独立于核对状态）：
  ┌──────┐     ┌──────┐     ┌──────────┐
  │ DIFF │ ──→ │ 处理  │ ──→ │ 已处理   │ ──→ 归档
  └──────┘     └──────┘     └──────────┘
```

状态说明：
- `UNMATCHED`: 未参与匹配（初始状态或取消匹配后）
- `MATCHED`: 匹配成功（自动配对或人工确认后）
- `PENDING`: 待确认（同金额多笔，需人工判断）
- `DIFF`: 差异（金额无匹配或金额不一致）
- `ARCHIVED`: 已归档（对账结果确认后）

## 匹配算法

### Step 1: 数据准备

```javascript
function prepareMatchingData(systemRecords, ledgerRecords) {
  // 1. 标准化
  systemRecords.forEach(r => {
    r.idCard = normalizeIdCard(r.idCard);
    r.insuranceType = normalizeInsuranceType(r.insuranceType);
  });
  ledgerRecords.forEach(r => {
    r.idCard = normalizeIdCard(r.idCard);
    r.insuranceType = normalizeInsuranceType(r.insuranceType);
    r.amount = roundToCents(r.amount);
  });

  // 2. 分组 key = idCard + '|' + insuranceType + '|' + billingMonth
  const systemGroups = groupByKey(systemRecords, makeGroupKey);
  const ledgerGroups = groupByKey(ledgerRecords, makeGroupKey);

  // 3. 找出所有 key
  const allKeys = new Set([...systemGroups.keys(), ...ledgerGroups.keys()]);

  return { allKeys, systemGroups, ledgerGroups };
}
```

### Step 2: 精确金额匹配

```javascript
function executeMatching(allKeys, systemGroups, ledgerGroups) {
  const results = {
    matched: [],    // 自动匹配成功
    pending: [],    // 待确认（同金额多笔）
    diffs: [],      // 差异
  };

  for (const key of allKeys) {
    const sysRecords = systemGroups.get(key) ?? [];
    const ledRecords = ledgerGroups.get(key) ?? [];

    // 统计金额出现次数
    const sysAmountMap = countByAmount(sysRecords);  // {635.25: [S1, S2], 800: [S3]}
    const ledAmountMap = countByAmount(ledRecords);  // {635.25: [T1, T2], 800: [T3]}

    const allAmounts = new Set([...sysAmountMap.keys(), ...ledAmountMap.keys()]);

    for (const amount of allAmounts) {
      const sysRecordsForAmount = sysAmountMap.get(amount) ?? [];
      const ledRecordsForAmount = ledAmountMap.get(amount) ?? [];

      if (sysRecordsForAmount.length > 0 && ledRecordsForAmount.length === 0) {
        // 系统有，台账无 → 差异（系统多）
        sysRecordsForAmount.forEach(r => {
          r.matchStatus = 'DIFF';
          r.diffType = 'system_more';
          r.diffAmount = r.amount;
          results.diffs.push({ systemRecord: r, ledgerRecords: [] });
        });

      } else if (sysRecordsForAmount.length === 0 && ledRecordsForAmount.length > 0) {
        // 台账有，系统无 → 差异（台账多）
        ledRecordsForAmount.forEach(r => {
          r.matchStatus = 'DIFF';
          r.diffType = 'ledger_more';
          r.diffAmount = -r.amount;
          results.diffs.push({ systemRecords: [], ledgerRecord: r });
        });

      } else if (sysRecordsForAmount.length === 1 && ledRecordsForAmount.length === 1) {
        // 唯一配对 → 自动匹配
        const sysRec = sysRecordsForAmount[0];
        const ledRec = ledRecordsForAmount[0];
        sysRec.matchStatus = 'MATCHED';
        sysRec.matchedLedgerId = ledRec.id;
        ledRec.matchStatus = 'MATCHED';
        ledRec.matchedSystemId = sysRec.id;
        // 回填：系统得到应缴月份，台账得到费用类型
        sysRec.payableMonth = ledRec.billingMonth;
        ledRec.feeTypeInferred = sysRec.feeType;
        results.matched.push({ systemRecord: sysRec, ledgerRecord: ledRec });

      } else {
        // 多笔同金额 → 待确认
        results.pending.push({
          amount,
          systemRecords: sysRecordsForAmount,
          ledgerRecords: ledRecordsForAmount,
        });
        sysRecordsForAmount.forEach(r => r.matchStatus = 'PENDING');
        ledRecordsForAmount.forEach(r => r.matchStatus = 'PENDING');
      }
    }
  }

  return results;
}
```

### Step 3: 金额不一致差异检测

Step 2 处理了"金额相同"的情况。对于同一分组内，系统侧和台账侧金额完全不同、无任何重合的情况（如系统只有 ¥150，台账只有 ¥142.50），双方都标记为"差异（金额不一致）"：

```javascript
function detectAmountMismatch(systemGroups, ledgerGroups) {
  const mismatches = [];

  for (const key of allKeys) {
    const sysRecords = systemGroups.get(key) ?? [];
    const ledRecords = ledgerGroups.get(key) ?? [];

    if (sysRecords.length === 0 || ledRecords.length === 0) continue;

    // 检查是否所有系统金额都没有台账匹配
    const sysAmounts = new Set(sysRecords.map(r => r.amount));
    const ledAmounts = new Set(ledRecords.map(r => r.amount));

    // 找出 unmatched 的系统记录（无相同金额台账）
    for (const sysRec of sysRecords) {
      if (!ledAmounts.has(sysRec.amount) && sysRec.matchStatus === 'UNMATCHED') {
        sysRec.matchStatus = 'DIFF';
        sysRec.diffType = 'amount_mismatch';
        // 找最接近的台账金额计算差额
        const closestLedger = findClosestAmount(ledRecords, sysRec.amount);
        sysRec.diffAmount = sysRec.amount - closestLedger.amount;
        mismatches.push({
          systemRecord: sysRec,
          closestLedger,
          diffAmount: sysRec.diffAmount,
        });
      }
    }

    // 找出 unmatched 的台账记录（无相同金额系统）
    for (const ledRec of ledRecords) {
      if (!sysAmounts.has(ledRec.amount) && ledRec.matchStatus === 'UNMATCHED') {
        ledRec.matchStatus = 'DIFF';
        ledRec.diffType = 'amount_mismatch';
      }
    }
  }

  return mismatches;
}
```

### 匹配结果统计

```javascript
function calculateStats(allSystemRecords, allLedgerRecords, results) {
  return {
    totalRecords: allSystemRecords.length + allLedgerRecords.length,
    matchedCount: results.matched.length,
    pendingCount: results.pending.length,
    diffCount: results.diffs.length,
    // 按类型细分
    systemMoreCount: results.diffs.filter(d => d.diffType === 'system_more').length,
    ledgerMoreCount: results.diffs.filter(d => d.diffType === 'ledger_more').length,
    amountMismatchCount: results.diffs.filter(d => d.diffType === 'amount_mismatch').length,
    // 金额统计
    matchedAmount: results.matched.reduce((sum, r) => sum + r.systemRecord.amount, 0),
    diffAmount: results.diffs.reduce((sum, d) => sum + Math.abs(d.systemRecord?.amount ?? 0), 0),
  };
}
```

### Tab 切换设计

页面采用 **Tab 切换** 方式分离系统侧和台账侧账单明细，解决两边混在一起无法辨识的问题。

**交互规则**：
- 点击 Tab 切换当前查看的账单侧，保留各自的筛选状态
- 已核对的记录 **同时出现在两个 Tab 中**，带跨引用标记
- 两侧 Tab 各自有独立的统计摘要、筛选栏和表格

## 导入流程

```
Step 1: 导入台账 Excel
   → 用户选择参保主体、参保规则、应缴月份
   → 上传 Excel 文件
   → 格式校验：
     - 必填列检查（姓名、身份证、险种、应缴月份、金额）
     - 身份证格式标准化（去空格、X大写）
     - 险种名称映射（别名 → 标准名称）
     - 金额格式检查（数值、保留2位小数）
   → 判重检查：参保主体 + 参保规则 + 应缴月份 是否已有台账
     - 有 → 弹窗提示"该维度下已有台账数据，是否覆盖？"
     - 无 → 继续
   → 导入预览：展示成功条数、错误条数、错误行详情
   → 用户确认导入

Step 2: 点击"开始对账"
   → 校验：是否已导入台账数据
   → 执行匹配算法（前端 JavaScript）
   → 匹配过程中：显示 loading 遮罩层 "正在对账..."
   → 完成后：
     - 刷新统计卡片
     - 刷新数据表格
     - Toast 提示 "对账完成：已核对 8 条，待确认 1 条，差异 2 条"

Step 3: 处理待确认和差异
   → 待确认：点击"确认核对"打开抽屉，智能推荐 + 手动调整
   → 差异：点击"详情"打开抽屉，查看分析，标记已处理
   → 可批量操作：
     - 全选 → 批量确认核对（仅对 PENDING 记录）
     - 全选 → 批量标记差异已处理（仅对 DIFF 记录）

Step 4: 归档对账结果
   → 校验：是否还有待处理和差异记录未处理
     - 有 → 弹窗提示"还有 N 条记录未处理，是否继续归档？"
     - 无 → 直接归档
   → 归档操作：
     - 将已核对记录的 match_status 写入数据库
     - 更新系统侧费用记录的 payable_month 字段
     - 标记台账记录和系统记录为 archived = true
   → Toast 提示 "归档完成：共归档 11 条记录"

Step 5: 归档后操作
   → 归档后数据锁定，不可修改
   → 如需修改：先取消归档，重新对账
   → 归档数据可用于后续统计报表
```

## 批量操作

批量操作在 **当前 Tab 的可见记录** 内生效：

| 操作 | 适用状态 | 行为 |
|------|----------|------|
| 批量确认核对 | PENDING | 对当前 Tab 中所有待确认记录执行智能推荐配对 |
| 批量标记差异已处理 | DIFF | 将当前 Tab 中所有差异记录标记为"已处理"状态 |
| 批量取消匹配 | MATCHED | 取消当前 Tab 中已核对记录的配对关系，恢复到 UNMATCHED 状态 |
| 全选 | 全部 | 仅选中当前 Tab 中可见的记录 |

## 边界情况

| 场景 | 处理 |
|------|------|
| 同一台账重复导入 | 参保主体+参保规则+应缴月份维度判重，提示是否覆盖 |
| 匹配成功后取消 | 可取消，恢复到 UNMATCHED 状态，重新进入匹配池 |
| 身份证格式不一致 | 导入时标准化（去空格、X大写） |
| 险种名称不一致 | 导入时映射，无法识别的标记"未知"，人工处理 |
| 对账后修改系统数据 | 归档后锁定，修改需先取消归档；未归档需重新对账 |
| 台账导入金额精度 | 系统侧和台账侧金额必须精确到分一致 |
| 同一应缴月份多笔同金额 | 标记"待确认"，智能推荐+批量映射 |
| 台账导入空文件 | 提示"文件中无有效数据"，不执行导入 |
| 台账导入部分字段为空 | 必填字段为空的行标记为格式错误，跳过该行 |
| 系统侧无匹配分组 | 台账有记录但系统侧无对应身份证/险种/月份，标记为"台账多"差异 |
| 台账侧无匹配分组 | 系统侧有记录但台账侧无对应身份证/险种/月份，标记为"系统多"差异 |
| 网络中断导致导入失败 | 提示"导入失败，请重试"，不清除已导入数据 |
| 大批量数据对账性能 | 超过 1000 条记录时显示进度条，分批执行匹配 |

## 错误处理

| 错误场景 | 错误信息 | 处理方式 |
|----------|----------|----------|
| Excel 格式不支持 | "不支持的文件格式，请使用 .xlsx 或 .xls 文件" | 拒绝导入 |
| Excel 列名不匹配 | "模板列名不匹配，请下载最新模板" | 拒绝导入 |
| 身份证校验失败 | "第 N 行身份证号格式不正确" | 标记错误行，跳过 |
| 金额格式错误 | "第 N 行金额格式不正确" | 标记错误行，跳过 |
| 应缴月份格式错误 | "第 N 行应缴月份格式不正确，请使用 YYYY-MM" | 标记错误行，跳过 |
| 判重冲突 | "参保主体+参保规则+应缴月份维度下已有台账数据" | 提示是否覆盖 |
| 匹配执行异常 | "对账过程中发生错误，请检查数据后重试" | 回滚核对状态 |
| 归档时数据不完整 | "还有 N 条记录未处理，是否继续归档？" | 用户确认后继续 |

  - 台账侧 Tab 的账单类型列：未匹配时显示 "—"，已核对时显示推断类型

### 实施差异记录

以下为实现过程中与初稿的细微差异：

1. **页面副标题**：从「统一账单模式 — 汇缴/补缴/调基补差合并展示（PRD v5）」更新为「Tab 切换模式 — 系统侧/台账侧分离展示（PRD v6）」
2. **统计卡片已移除**：全局统计卡片（网格和单行条）均已移除，不再占用页面垂直空间
3. **金额整合到筛选按钮**：统计金额整合到核对状态筛选按钮中显示（如「全部 ¥3386.25」「已核对 ¥1500.00」），两侧各自独立
4. **摘要条已移除**：系统侧和台账侧的 `.summary-strip` 已完全移除
5. **筛选栏**：系统侧和台账侧提供「核对状态」（全部/已核对/待确认/差异，带金额）+「账单类型」（全部/汇缴/补缴/调基补差）两组筛选
6. **`filterRecords` 函数**：顶部搜索按钮的原引用函数已删除，新增桩函数调用 `renderAllTables()`，后续可扩展为实际按月份/结算主体/姓名过滤
7. **批量操作**：系统侧和台账侧各自独立的全选/确认/取消逻辑，互不影响
