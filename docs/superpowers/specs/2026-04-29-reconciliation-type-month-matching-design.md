# 社保对账复核：类型/月份互推匹配设计

## Context

青阳云系统根据员工参保异动生成应缴费用，包含汇缴、补缴、调基补差三种类型。系统知道费用类型，但补缴和补差不知道应缴月份。税务台账（Excel导入）知道应缴月份，但不知道费用类型。

核心矛盾：系统有类型无月份，台账有月份无类型。双方共同拥有：身份证、险种、金额。

同一应缴月份内，一个员工同一险种可能产生多笔费用：1笔汇缴 + N笔补缴 + M笔补差。台账中同样有多笔，但无法区分类型。

金额特征（零头、特殊值）无法作为类型判断依据。同一金额在系统侧和台账侧可能各有多笔，无法确定对应关系。

## 约束确认

| 项目 | 确认值 |
|------|--------|
| 匹配范围 | 身份证 + 险种 + 应缴月份 |
| 匹配方式 | 精确金额匹配（一分不差） |
| 触发方式 | 台账导入后，用户手动点击"开始对账" |
| 匹配成功后 | 系统回填应缴月份，台账回填费用类型 |
| 同金额多笔 | 标记"待确认"，人工判断 |
| 待确认交互 | 智能推荐 + 批量映射 + 确认 |
| 对账输出 | 匹配成功归档 + 差异处理流程 |
| 判重规则 | 参保主体 + 参保规则 + 应缴月份 维度只保留一份台账 |
| 差异类型 | 系统多（台账无）、台账多（系统无）、金额不一致（双方都有但金额不同） |

### 台账侧记录（新增字段）

### 导入模板格式

Excel 必填列：姓名、身份证、险种、应缴月份、金额

导入时做格式校验：
- 身份证格式标准化（去空格、X大写）
- 险种名称映射（如"基本养老保险"→"养老"），无法识别的标记为"未知"

## 匹配算法

### Step 1: 分组

按 `身份证 + 险种 + 应缴月份` 将系统侧费用和台账记录分组。每个组是一个独立的匹配单元。

### Step 2: 精确金额匹配

在同一组内，统计系统侧和台账侧各金额的出现次数：

```
for each group:
  systemAmounts = countByAmount(group.systemRecords)
  ledgerAmounts = countByAmount(group.ledgerRecords)

  for each amount in systemAmounts:
    sysCount = systemAmounts[amount]
    ledCount = ledgerAmounts[amount] ?? 0

    if ledCount == 0:
      // 系统有，台账无 → 差异（系统多）
      markAllDiff(systemRecords with this amount)
    else if sysCount == 1 and ledCount == 1:
      // 唯一配对 → 自动匹配
      matchPair(sysRecord, ledgerRecord)
      fillCrossInfo(sysRecord, ledgerRecord)  // 回填类型和月份
    else:
      // 多笔同金额 → 待确认
      markAllPending(systemRecords + ledgerRecords with this amount)

  // 检查台账侧多余记录
  for each amount in ledgerAmounts:
    if amount not in systemAmounts:
      markAllDiff(ledgerRecords with this amount)  // 台账多
```

### Step 3: 金额不一致差异检测

Step 2 只处理精确金额匹配。对于同一分组内，系统侧和台账侧都有记录但金额完全不同的情况（如系统只有 ¥150，台账只有 ¥142.50），双方都标记为"差异（金额不一致）"，计算差额（¥150 - ¥142.50 = +¥7.50），在详情中展示可能原因。

### 匹配结果分类

| 状态 | 含义 | 处理 |
|------|------|------|
| MATCHED | 金额唯一配对成功 | 直接归档，回填类型和月份 |
| PENDING | 同金额多笔，无法确定配对 | 进入待确认，人工判断 |
| DIFF | 金额无匹配或金额不一致 | 进入差异处理 |

## 导入流程

```
1. 导入台账 Excel
   → 格式校验（身份证标准化、险种映射）
   → 判重检查（参保主体 + 参保规则 + 应缴月份）
   → 重复导入提示"是否覆盖已有台账数据"

2. 导入预览
   → 展示导入成功条数、格式错误条数
   → 台账数据以原始状态展示，match_status 初始为 null

3. 点击"开始对账"
   → 系统执行精确金额匹配
   → 匹配过程中显示 loading 状态
   → 完成后刷新统计卡片和表格

4. 查看结果
   → 已匹配：自动归档，回填类型和月份
   → 待确认：点击"确认配对"打开抽屉，人工选择配对
   → 差异：点击"详情"打开抽屉，查看原因分析，标记已处理

5. 批量归档
   → 所有待处理和差异记录处理完毕后，点击"归档对账结果"
   → 已匹配记录进入归档状态，用于后续统计和报表
   → 未处理的差异记录保留为"待处理"状态
```

## 边界情况

| 场景 | 处理 |
|------|------|
| 同一台账重复导入 | 参保主体+参保规则+应缴月份维度判重，提示是否覆盖 |
| 匹配成功后取消 | 可取消，恢复到未匹配状态，重新进入匹配池 |
| 身份证格式不一致 | 导入时标准化（去空格、X大写） |
| 险种名称不一致 | 导入时映射，无法识别的标记"未知"，人工处理 |
| 对账后修改系统数据 | 归档后锁定，修改需先取消归档；未归档需重新对账 |
| 台账导入金额精度 | 系统侧和台账侧金额必须精确到分一致 |
| 同一应缴月份多笔同金额 | 标记"待确认"，智能推荐+批量映射 |
