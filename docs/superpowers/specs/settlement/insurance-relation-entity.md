# 结算主体减员权限与费用管理设计

> 日期：2026-04-25
> 状态：待评审

## 背景

当前系统员工调动后，只有新结算主体能操作减员。业务要求原结算主体也能对其曾经管理的员工进行减员操作，无论保险归属于哪个参保主体。

### 核心概念

| 维度 | 含义 | 变化规则 |
|------|------|---------|
| **结算主体** | 员工的管理归属 + 财务归属 | 随调动/重新入职转移 |
| **参保主体** | 保险关系的归属（实际缴纳社保的主体） | 不随结算主体变化 |

### 行业特征

人力资源服务行业普遍存在「代缴」模式：结算主体与参保主体不一致。员工由 A 主体管理，但社保由 C 主体代缴。

### 当前系统缺陷

员工调动后，系统只允许新结算主体操作减员。原结算主体看不到该员工，无法操作停保/减员，导致：
1. 社保减员延迟，产生不必要的费用
2. 原结算主体无法履行管理责任
3. 财务对账数据断裂

## 设计目标

1. 原结算主体能查询并减员曾经管理的员工
2. 保险关系绑定参保主体，不随员工调动转移
3. 支持结算主体与参保主体不一致的代缴模式
4. 合并付款申请以发起的结算主体为财务归属
5. 完整的费用双层管理模型（缴费层 + 对账层）

## 查询逻辑

### 员工参保档案列表查询

```sql
-- 显示当前结算主体管理的员工 + 有历史管理关系的员工
SELECT DISTINCT e.* FROM employee e
WHERE e.settlement_entity_id = #{current_entity_id}
   OR e.employee_id IN (
     SELECT employee_id FROM settlement_history 
     WHERE settlement_entity_id = #{current_entity_id}
       AND end_date IS NOT NULL
   )
```

### 筛选增强

新增「员工归属」筛选条件：

| 值 | 含义 | SQL 条件 |
|----|------|----------|
| 全部 | 当前 + 历史 | 原查询不变 |
| 当前所属 | 仅当前员工 | `employee.settlement_entity_id = #{current_entity_id}` |
| 已调出 | 仅历史员工 | `employee.settlement_entity_id != #{current_entity_id} AND EXISTS(...)` |

## 操作权限

### 减员权限

| 主体类型 | 减员权限 | 范围 |
|---------|---------|------|
| 当前结算主体 | ✅ | 所有保险 |
| 原结算主体 | ✅ | 所有保险（无论参保主体是谁） |

**规则**：减员操作基于「曾管理关系」而非仅「当前管理关系」。原结算主体可以直接操作减员，不需要走审批流程。

### 增员权限

| 主体类型 | 增员权限 | 范围 |
|---------|---------|------|
| 当前结算主体 | ✅ | 新建保险关系 |
| 原结算主体 | ❌ | 不允许新增 |

## 资金流

### 完整流程

```
1. 费用产生
   员工参保 → 费用归属结算主体 X

2. 付款申请
   结算主体 X 发起付款申请 → 向参保主体 A 申请付款给社保局

3. 统一缴费
   参保主体 A 汇总所有申请 → 向社保局统一缴纳

4. 内部对账
   社保局回执到参保主体 A → 费用明细按结算主体拆分 → 各结算主体向客户收款
```

### 付款申请模式

**标准模式**（按结算主体分别申请）：
```
参保主体 A 收到：
  - 结算主体 X 申请：60,000 元
  - 结算主体 Y 申请：30,000 元
  - 结算主体 Z 申请：10,000 元
```

**合并模式**（一个结算主体代申请）：
```
结算主体 X 发起合并付款申请：100,000 元
  ├─ 本主体应收：60,000 元
  ├─ 结算主体 Y 费用：30,000 元（内部划转）
  └─ 结算主体 Z 费用：10,000 元（内部划转）

对外：参保主体 A 只认 X 的 100,000 元申请
对内：Y、Z 的费用通过 X 内部结算
```

### 费用双层模型

**缴费层**（参保主体维度）：
```
参保主体 A | 2026-04 | 应缴 100,000 | 已缴 100,000 | ✅ 已对平
```

**对账层**（结算主体维度）：
```
结算主体 X | 2026-04 | 应收 60,000 | 已收 60,000 | ✅ 已结清
结算主体 Y | 2026-04 | 应收 30,000 | 已收 20,000 | ⚠️ 欠费 10,000
```

### 核心原则

1. **缴费以参保主体为准** — 对外（社保局）的唯一出口
2. **对账以结算主体为准** — 对内（客户）的收费依据
3. **两条线独立管理，通过费用明细关联** — 任何一笔费用都能追溯到「谁交的、该收谁的」

## 调动/重新入职逻辑

### 调动

- **只改变** `employee.settlement_entity_id`
- **不改变** `employee.insurance_entity_id` 和保险关系
- 写入 `settlement_history` 记录
- 新主体如需为该员工参保，需新建保险关系

### 重新入职

- 视为全新员工，生成新的 `employee_id`
- 原保险记录保留在原参保主体
- 新主体重新参保

## 统计面板

统计面板（已参保/未参保/已停保）的计算范围：

- 仅统计 `insurance_relation.insurance_entity_id = #{current_entity_id}` 的记录（当前主体为参保主体）
- 或统计 `settlement_history` 中当前结算主体有管理关系的记录

## 影响范围

### 受影响页面/功能

- 员工参保档案列表
- 保险关系新增/编辑/停保操作
- 员工调动逻辑
- 统计面板计算
- 高级搜索（新增「员工归属」条件）
- 付款申请（新增合并付款模式）
- 费用明细（新增双层管理）

### 不受影响

- 保险台账
- 参保规则配置
- 二级户管理

## 数据迁移

### 迁移步骤

1. 将现有 `employee.insurance_entity_id` 从历史参保数据中恢复
2. 建立 `settlement_history` 表，从异动记录中提取历史结算关系
3. 迁移脚本需支持回滚

### 兼容性

- 迁移期间，新旧查询逻辑并存
- 前端增加「员工归属」筛选条件，但不影响现有查询

## 验收标准

1. 员工调动后，原结算主体仍能查询到该员工
2. 原结算主体可以对其曾管理的员工进行减员操作
3. 列表合并显示当前所属和已调出员工，已调出员工加标识
4. 「员工归属」筛选条件正常工作
5. 统计面板计算范围正确
6. 付款申请支持合并模式，费用归属到发起主体
7. 费用明细支持双层对账（缴费层 + 对账层）
8. 数据迁移脚本正确执行
