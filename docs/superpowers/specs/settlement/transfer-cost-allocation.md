# 员工调动后费用归属调整设计

## Context

青阳云HRO系统的保险福利核算模块中，员工费用按项目/客户/合同归属计算。当员工发生调动或重新入职后，其归属关系发生变化，但已生成的账单仍挂在原归属名下，导致费用归属错误。

当前系统中异动申报功能已存在，包含调动、重新入职等类型，审批通过后员工信息中的"所属客户/合同/项目"字段会更新。但系统没有检测和处理已生成账单归属错误的逻辑。

## 核心场景

### 场景一：员工调动后费用归属变更

员工从项目A调动到项目B后，整个月的费用应归属到项目B，但已生成的账单仍在A名下。通过重新核算账单来更新费用归属。

同一员工允许参保后一次性核算多个月份账单，不同月份可能归属不同项目，账单需要按客户/合同/项目进行拆分。

**设计原因**：费用归属反映的是员工在某一时刻的项目归属关系。员工调动后，其劳动关系、服务关系、成本承担方已转移到新归属方，因此已生成但未归档的账单应按当前归属重新核算。未归档账单代表尚未最终确认的费用，具备调整条件。

### 场景二：政策调整补差补退费用归属

年度政策调整（如社保基数上下限变更、缴费比例调整）时，系统会对历史月份产生补差补退费用。此时补差补退费用应按历史月份的原费用归属进行分配——**向谁收费，就向谁退费**。

例如：2026-03月份费用归属为"人才产业园"，同年7月发生政策调整产生3月补差费用，该补差费用仍归属"人才产业园"，不因员工后续调动而改变。

**设计原因**：补差补退费用是原始费用的"修正项"，与原始费用存在强绑定关系。原费用向谁收取，其修正产生的补差或退费自然应归属同一方。如果将补差补退费用归属到当前归属方，会导致历史费用的责任方与实际承担方错位——原归属方享受了错误的费率却不承担修正后果，当前归属方未享受原费率却承担了修正成本，这在财务上不合理。

### 场景三：手动补缴历史月份费用归属

员工在A项目参保期间某月份未缴，后调动到B项目。在B项目下手动补缴A项目期间的费用时，补缴费用归属**补缴月份的原归属方**（A项目），而非当前归属方（B项目）。

例如：4月在A项目参保但未缴，6月调动到B项目，用户在B项目下为4月单独补缴。该补缴费用归属A项目，因为4月时员工归属为A项目。

**设计原因**：补缴费用是对历史未缴行为的补正，其本质是"本应在那个时间段缴纳但未缴的费用"。费用发生的期间归属方才是责任承担方，而非操作行为的发起方。如果不按补缴月份归属，会出现A项目漏缴的费用由B项目承担的错位，导致项目间成本错配和内部结算纠纷。

### 场景四：追溯参保产生历史费用

员工在A项目期间未参保，后调动到B项目，在B项目参保时生效月份追溯至A项目期间。此时产生的历史月份（追溯期间）费用归属**当前参保方**（B项目），但需在账单中明确标注"补缴历史月份费用（原归属A项目）"。

例如：员工3月在A项目入职但未参保，5月调动到B项目并在B项目参保，生效月份追溯至3月。3-4月补缴费用归属B项目，但账单标注原归属信息，便于B项目与A项目间内部结算。

**设计原因**：与场景三的区别在于，追溯参保是由当前归属方（B项目）主动发起的参保行为，参保动作的系统记录归属于B项目。但追溯期间（3-4月）员工实际在A项目工作，费用责任应归A项目。因此账单归属设为B项目（参保发起方），同时标注原归属信息，作为A、B项目间内部结算和费用转移的依据。如果不标注原归属，B项目将承担本应由A项目负责的费用。

### 场景五：审批通过日与生效日不一致

员工调动审批在6月通过，但生效日追溯至4月。系统应在**审批通过时**触发检测，检测范围为**生效日起至当前月份**的所有未归档账单。

例如：员工4月在A项目，6月审批通过调动至B项目，生效日设为4月。检测范围包括4月、5月、6月的账单，4月及之后的归属应为B项目。

**已归档账单处理**：若生效日至审批日之间有账单已归档，已归档账单不受影响，仅调整未归档部分。例如4月账单已归档、5月未归档，则仅调整5月及之后的归属。归档账单的归属变更需通过财务调账流程处理。

**设计原因**：检测时机以审批通过为触发点，因为只有审批通过后系统才确认归属变更生效。检测范围以生效日为起点，因为生效日才是费用归属关系实际发生变更的时间点。如果仅从审批通过当月检测，会漏掉生效日至审批月之间已生成的账单。已归档账单代表财务已确认结算，不在常规费用调整范围内。

### 场景六：离职后重新入职且未停保

员工在A项目工作期间未办理停保即离职，之后以新员工身份入职B项目。离职前产生的费用继续由A项目承担，离职后产生的费用由B项目承担。

例如：员工3月在A项目离职未停保，5月入职B项目。3-4月产生的费用由A项目承担，5月起由B项目承担。

**设计原因**：停保是费用归属的责任分界点。若未停保，费用持续产生，离职前期间员工劳动关系仍在A项目，费用由A项目承担。入职B项目后，劳动关系和参保关系均转移至B项目，此后费用由B项目承担。此场景的归属逻辑与场景四（追溯参保）不同：场景四是B项目参保追溯至A项目期间，费用归B但标注原归属；场景六是明确停保为分界点，离职前后各归原方。

### 关键原则总结

| 场景 | 费用归属 | 依据 |
|------|------|------|
| 调动后重新核算已生成账单 | 按员工当前归属更新 | 账单记录归属与员工当前归属不一致 |
| 政策调整补差补退 | 追溯到历史月份原归属方 | 费用属于哪个月，就归属该月的原归属方 |
| 手动补缴历史月份 | 按补缴月份原归属方 | 补缴是对过去某月未缴的补正 |
| 追溯参保产生历史费用 | 按当前参保方（新归属） | 参保动作由当前归属方发起，但标注原归属便于内部结算 |
| 审批通过日与生效日不一致 | 以生效日为起点检测 | 生效日是归属关系实际变更的时间点 |
| 离职后重新入职未停保 | 停保前归离职方，入职后归新方 | 停保是费用归属的责任分界点 |

### 预留场景（本期不实现）

| 场景 | 说明 | 预留方案 |
|------|------|------|
| 月中调动按比例拆分 | 员工当月15日调动，当月费用归属按比例分配 | 检测机制支持按日粒度记录归属变更时间线，费用调整面板可扩展"按比例拆分"操作 |

## 设计决策

### 方案选择

**方案 B：独立"费用调整"面板 + 批量重算**

在保险福利核算页面顶部显示通知横幅，点击打开右侧费用调整面板，集中管理所有因调动产生的费用调整事项，支持批量操作和查看调整明细预览。

不选择方案A（太轻量，无预览）和方案C（审批流程过度设计）。

### 检测机制

| 项目 | 场景一：调动/重新入职 | 场景二：政策调整补差补退 | 场景三：手动补缴历史月份 | 场景四：追溯参保 | 场景五：审批生效日不一致 | 场景六：离职未停保 |
|------|------|------|------|------|------|------|
| **触发点** | 员工"所属客户"、"所属合同"、"所属项目"任一字段变更 | 年度政策调整生效、社保/公积金基数上下限变更、缴费比例变更 | 用户手动为员工补缴历史月份费用 | B项目参保时生效月份追溯至历史期间 | 异动审批通过且生效日早于审批通过日 | 员工离职未停保 + 以新员工身份入职新归属方 |
| **检测范围** | 所有未归档账单 | 所有未归档账单中的补差补退费用记录 | 所有未归档账单中的补缴费用记录 | 追溯期间产生的所有费用记录 | 生效日至当前月份的所有未归档账单 | 离职前产生的所有未停保费用记录 |
| **检测逻辑** | 查询该员工关联的所有未归档账单，对比账单中记录的归属字段与员工当前归属字段，若不一致则标记为"待处理" | 查询补差补退费用对应的历史月份，获取该月份原始费用归属，将补差补退费用归属到同一归属方 | 补缴费用归属按补缴月份的原归属方确定，不因员工当前归属而改变 | 追溯期间费用归属设为当前参保方，账单标注原归属信息 | 以生效日为起点，检测生效日至当前月份的未归档账单归属，已归档账单不受影响 | 以停保状态为分界点，离职前费用由离职方承担，入职后费用由新方承担 |

每个险种占两行：
- **第一行（申报行）**：参保基数、应缴金额、固定存缴额 → 显示按参保基数计算的应缴金额
- **第二行（承担行）**：承担基数、实担金额 → 显示实际承担的基数和金额

关键逻辑：
- 单位实担 = 承担基数 × 缴纳比例
- 个人实担 = 个人应缴 + (单位应缴 - 单位实担)，即个人承担单位少缴的部分
- 参保基数 = 承担基数时，应缴 = 实担（正常情况）
- 个人无缴费的险种（如工伤）个人列显示 `—`
- 合计行底部展示：单位应缴合计、单位实担合计、个人应缴合计、个人实担合计、总计

#### 5. 错误处理

| 场景 | 处理 |
|------|------|
| 重算中 | 行状态显示"重算中..."，按钮禁用 |
| 重算成功 | 记录移入"已处理"Tab，顶部Toast"重算成功" |
| 重算失败 | 记录保留在"待处理"Tab，显示红色错误标签和失败原因 |
| 账单已归档 | 不显示（归档账单不受影响） |

## 数据流

```
场景一：员工调动/重新入职 → 员工归属字段变更 → 自动检测未归档账单 → 生成待调整记录
场景二：政策调整生效 → 补差补退费用产生 → 查询历史月份原始归属 → 生成待调整记录
场景三：手动补缴历史月份 → 补缴操作完成 → 按补缴月份原归属方标记 → 生成待调整记录
场景四：追溯参保 → B项目参保生效 → 追溯期间费用归属新方+标注原归属 → 生成待调整记录
场景五：异动审批通过 → 以生效日为起点检测未归档账单 → 生成待调整记录
场景六：离职未停保 → 以停保为分界点检测费用归属 → 生成待调整记录
       ↓
保险福利核算页面顶部横幅通知
       ↓
用户点击"查看详情" → 打开费用调整面板
       ↓
用户预览/重算 → 触发账单重新核算
       ↓
账单按归属拆分 → 员工明细列表更新 → 记录移入"已处理"
```

  - 保险福利核算页面（`/salaryManage/insurance/accountingBill/index`）— 新增通知横幅
  - 账单明细页面（`/accountingBill/billDetailPage`）— 员工列表拆分行
  - 员工费用详情页面（`/accountingBill/employeeDetailPage`）— 重构为单一列表

## 验证方式

1. 在系统中创建员工调动记录（所属项目变更）
2. 进入保险福利核算页面，验证顶部出现橙色通知横幅
3. 点击"查看详情"，验证费用调整面板打开，列表中显示该员工及归属变更信息
4. 点击"预览"，验证弹窗中月份→归属映射正确
5. 点击"重算"，验证账单按归属拆分成功
6. 进入员工费用详情页面，验证费用明细列表按归属分组展示
