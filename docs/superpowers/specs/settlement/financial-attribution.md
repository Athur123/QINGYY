# 结算主体维度财务归属与业务追溯设计

> 日期：2026-04-25
> 状态：待评审

## 背景

### 问题概述

结算主体在系统设计之初仅作为员工的一个管理属性（数据过滤条件），未考虑按结算主体进行严格的业务隔离和财务隔离。但在实际生产使用和对外宣传中，客户按结算主体进行业务划分与财务划分，导致以下核心问题：

1. **历史记录丢失**：员工调出后，原结算主体无法追溯其离职、合同、异动、参保等历史记录
2. **费用归属错误**：员工调动后，历史未结算费用和后续退费随员工属性变更转移到新主体，导致账单出错
3. **财务无法对账**：付款申请、应收费用、结算明细均未按结算主体固化，导致成本、收入无法归属到正确主体

### 核心矛盾

| 维度 | 系统设计 | 客户实际使用 |
|------|---------|-------------|
| 结算主体 | 员工的一个属性字段 | 业务和财务的隔离边界 |
| 员工调动 | 属性变更，历史记录跟随 | 历史记录应留在原主体 |
| 费用归属 | 跟随员工当前属性 | 跟随产生费用时的属性 |

## 设计目标

1. **员工档案版本化**：关键信息变更产生新版本，业务操作记录当时版本号
2. **费用归属固化**：费用明细在产生时固化结算主体/客户/合同/项目，不随员工调动变更
3. **业务权限矩阵**：明确各模块对已调出员工的查看和操作权限
4. **财务账单固化**：账单生成后成员明细锁定，不随员工调动变更

## 数据模型

### 员工档案版本表

```
employee_archive_version:
  - version_id          版本号
  - employee_id         员工ID
  - settlement_entity_id 结算主体
  - customer_id         所属客户
  - contract_id         所属合同
  - project_id          所属项目
  - insurance_entity_id  参保主体
  - start_date          生效开始日期
  - end_date            生效结束日期
  - change_reason       变更原因（调动/重新入职/信息变更）
```

每次员工调动或重新入职时，关闭当前版本（设置 end_date），创建新版本。

### 业务操作快照

所有业务操作记录操作时的员工档案版本：

```
business_operation:
  - operation_id        操作ID
  - employee_id         员工ID
  - archive_version_id  操作时的档案版本号
  - operation_type      操作类型（增员/减员/合同签署/离职等）
  - operation_date      操作时间
  - operator_id         操作人
  - settlement_entity_id 操作时的结算主体（快照）
  - ...                 业务相关字段
```

### 费用明细表

```
cost_detail:
  - cost_id             费用唯一标识
  - employee_id         员工ID
  - archive_version_id  产生费用时的档案版本号
  - settlement_entity_id 费用归属的结算主体（固化）
  - customer_id         费用归属的客户（固化）
  - contract_id         费用归属的合同（固化）
  - project_id          费用归属的项目（固化）
  - insurance_entity_id  参保主体
  - insurance_type      险种名称
  - amount              费用金额
  - month               费用月份
  - status              应收/已收/欠费/退费
  - created_date        产生时间
```

**关键**：费用明细一旦生成，`settlement_entity_id`、`customer_id`、`contract_id`、`project_id` 不可变更。即使员工后续调动，费用归属不变。

### 账单表

```
bill:
  - bill_id             账单ID
  - settlement_entity_id 结算主体
  - customer_id         客户
  - month               账单月份
  - total_amount        总金额
  - status              待确认/已确认/已结清
  - created_date        生成时间

bill_detail:
  - detail_id           明细ID
  - bill_id             关联账单
  - cost_id             关联费用
  - employee_id         员工ID
  - archive_version_id  费用产生时的版本号（固化）
  - settlement_entity_id 结算主体（固化）
  - customer_id         客户（固化）
  - contract_id         合同（固化）
  - project_id          项目（固化）
  - amount              金额
```

**关键**：账单生成后，明细锁定。员工调动不影响已生成的账单。

### 付款申请表

```
payment_request:
  - payment_id          付款申请ID
  - settlement_entity_id 发起申请的结算主体
  - insurance_entity_id  收款方参保主体
  - total_amount        付款总金额
  - status              待审批/已审批/已缴费
  - merge_flag          是否合并付款
  - created_date        申请时间

payment_request_detail:
  - detail_id           明细ID
  - payment_id          关联付款申请
  - cost_id             关联费用
  - settlement_entity_id 费用归属的结算主体（固化）
  - amount              金额
```

## 费用归属固化规则

### 正常费用

费用产生时，按操作时的员工档案版本固化归属：

```
2026-03: 员工张三在结算主体A参保 → 费用归属A
2026-04: 张三调动到结算主体B → 3月费用仍归属A，不受影响
2026-04: 张三在结算主体B新参保 → 4月费用归属B
```

### 退费/补收补退

退费按原费用产生时的归属处理，不随员工当前属性变化：

```
原费用：2026-03 归属A的5000元社保费
退费：2026-05 产生退费 → 退费归属A，归属到原客户/合同/项目
```

### 差额管理

差额按费用产生时的归属处理：

```
原费用：2026-03 归属A的5000元
差额补收：2026-05 补收500元 → 归属A，不归属到员工当前所在主体
```

### 合并付款申请

合并付款申请时，财务归属到发起的结算主体：

```
结算主体X发起合并付款申请100,000元：
  - 本主体费用：60,000元（归属X）
  - 结算主体Y费用：30,000元（内部划转，归属X）
  - 结算主体Z费用：10,000元（内部划转，归属X）

对内：通过内部结算记录追溯Y、Z的费用
对外：参保主体只认X的申请
```

## 业务与财务隔离矩阵

### 隔离类型定义

| 类型 | 含义 | 核心机制 |
|------|------|---------|
| **业务隔离** | 数据按结算主体可见，历史记录可追溯 | 档案版本化 + 权限矩阵 |
| **财务隔离** | 费用归属固定，不随员工属性变化 | 费用快照 + 账单固化 |

### 第一类：纯业务隔离（无直接费用）

| 模块 | 业务隔离 | 财务隔离 | 说明 |
|------|---------|---------|------|
| 花名册 | ✅ | ❌ | 历史记录可查，状态按版本判断 |
| 异动管理 | ✅ | ❌ | 入职/转正/调动/离职记录按版本归属 |
| 合同管理 | ✅ | ❌ | 合同台账按版本归属，调出前合同可查看 |
| 人事证明 | ✅ | ❌ | 证明按出具时版本生成 |
| 考勤管理 | ✅ | ❌ | 考勤记录按月份归属 |
| 工资条 | ✅ | ❌ | 历史工资条按版本展示 |

**隔离方式**：所有操作记录当时的档案版本号，查询时按结算主体 + 时间范围过滤对应版本的数据。原主体可查看历史记录，不可修改（除特殊场景）。

### 第二类：业务隔离 + 弱财务隔离

| 模块 | 业务隔离 | 财务隔离 | 说明 |
|------|---------|---------|------|
| 薪资档案 | ✅ | ⚠️ | 薪资标准按版本，但调出后新主体重新定薪 |
| 社保待遇 | ✅ | ⚠️ | 待遇记录按版本，影响较小 |
| 年度缴费工资申报 | ✅ | ⚠️ | 申报按版本，影响次年基数 |
| 社保台账明细 | ✅ | ⚠️ | 台账按版本记录，作为费用计算依据 |
| 差额管理 | ✅ | ✅ | 差额归属到原费用主体，不随调动转移 |

**隔离方式**：同第一类 + 费用相关记录按产生时版本固化归属，退费/补收补退归属到原费用主体。

### 第三类：强业务隔离 + 强财务隔离（核心）

| 模块 | 业务隔离 | 财务隔离 | 说明 |
|------|---------|---------|------|
| 员工参保档案 | ✅ | ✅ | 减员/补缴/调基权限需扩展至原主体 |
| 保险福利核算 | ✅ | ✅ | 费用按产生时主体固化，不随调动转移 |
| 异动申报 | ✅ | ✅ | 申报记录按版本归属，原主体可减员 |
| 结算明细 | ✅ | ✅ | 费用按产生时客户/合同/项目取值 |
| 付款申请 | ✅ | ✅ | 申请归属到发起主体，合并付款内部划转 |
| 应收单 | ✅ | ✅ | 应收按费用归属固化，账单锁定后不可变更 |
| 工资发放 | ✅ | ✅ | 发放记录按版本归属，成本归属清晰 |

**隔离方式**：费用产生时完整快照结算主体/客户/合同/项目；账单生成后明细锁定；合并付款申请财务归属到发起主体；退费/补收补退按原费用归属处理。

## 员工档案版本化规则

### 触发版本化的操作

| 操作 | 是否版本化 | 说明 |
|------|-----------|------|
| 调动 | ✅ | 结算主体/客户/合同/项目变化 |
| 重新入职 | ✅ | 视为新员工，全新版本 |
| 客户变更 | ✅ | 客户变化 |
| 合同变更 | ✅ | 合同/项目变化 |
| 基本信息变更（姓名/电话等） | ❌ | 不影响财务归属 |

### 版本查询

```sql
-- 查询某结算主体下的员工档案
SELECT e.*, v.* 
FROM employee e
JOIN employee_archive_version v ON e.employee_id = v.employee_id
WHERE v.settlement_entity_id = #{current_entity_id}
  AND v.start_date <= #{query_date}
  AND (v.end_date IS NULL OR v.end_date >= #{query_date})
```

## 实施优先级

### 第一阶段：费用归属固化（最高优先级）

1. 费用明细表增加 `archive_version_id`、`settlement_entity_id`、`customer_id`、`contract_id`、`project_id` 固化字段
2. 费用生成时快照员工档案属性
3. 账单生成时固化明细，不可变更
4. 退费/补收补退按原费用归属处理

### 第二阶段：员工档案版本化

1. 建立 `employee_archive_version` 表
2. 现有员工数据迁移，补全历史版本
3. 调动/重新入职等操触发版本创建
4. 业务操作记录档案版本号

### 第三阶段：业务权限矩阵落地

按模块逐个实施查看和操作权限：

1. 核心人事（花名册、异动、合同）
2. 保险福利（参保档案、核算、待遇、申报）
3. 上下游（结算、工资条）
4. 财务管理（付款申请、应收、工资发放）

## 数据迁移

### 迁移步骤

1. 分析现有员工数据，提取历史调动记录
2. 建立 `employee_archive_version` 表，补全历史版本
3. 对已有费用明细，按产生时间回溯固化归属
4. 对已有账单，固化明细，不可变更
5. 迁移脚本需支持回滚

### 兼容性

- 迁移期间，新旧数据模型并存
- 前端增加「员工归属」筛选条件
- 财务模块增加「费用归属」筛选条件

## 影响范围

### 受影响模块

- 核心人事：花名册、异动管理、合同管理
- 智能薪酬：薪资档案
- 保险福利：参保档案、福利核算、社保待遇、异动申报、社保台账、差额管理
- 上下游：结算明细、工资条
- 财务管理：付款申请、应收单、工资发放

### 不受影响

- 参保规则配置
- 二级户管理
- 系统设置

## 验收标准

### 费用归属

1. 员工调动后，历史费用归属不变
2. 退费/补收补退归属到原费用主体
3. 账单生成后明细锁定，不受员工调动影响
4. 合并付款申请归属到发起主体

### 员工档案版本

1. 每次调动/重新入职生成新版本
2. 业务操作记录当时版本号
3. 按版本可追溯历史状态

### 业务权限

1. 各模块按权限矩阵实现查看和操作控制
2. 原结算主体可查看历史，不可修改（除允许操作的场景）
3. 新结算主体可新增记录

### 数据迁移

1. 历史数据正确迁移
2. 迁移脚本支持回滚
3. 新旧数据模型兼容运行
