# 结算主体维度财务归属 - 阶段一 PRD

> 日期：2026-04-25
> 状态：待开发

## 背景

当前系统结算主体仅作为员工的一个属性字段（数据过滤条件），未考虑按结算主体进行严格的业务隔离和财务隔离。实际生产中，客户按结算主体进行业务划分与财务划分，导致历史记录丢失、费用归属错误、财务无法对账等问题。

详见设计文档：[2026-04-25-settlement-entity-financial-attribution-design.md](./2026-04-25-settlement-entity-financial-attribution-design.md)

## 验收标准

### 数据库

1. [ ] 迁移脚本可在测试环境执行成功，无语法错误
2. [ ] 历史数据迁移正确，员工版本记录数与异动记录一致
3. [ ] 回滚脚本执行后，数据库恢复到迁移前状态
4. [ ] 新增索引生效，查询性能无明显下降

### 交互原型

1. [ ] 员工档案版本管理页面可通过浏览器访问
2. [ ] 支持按结算主体、状态筛选版本
3. [ ] 版本时间线正确展示，差异高亮准确
4. [ ] 费用归属固化演示三个场景可切换查看
5. [ ] 费用明细表支持按归属主体筛选

## 关键设计原则

### 费用归属固化

费用明细在产生时固化结算主体/客户/合同/项目，后续员工调动不影响已生成费用的归属。

```
2026-03: 张三在结算主体A参保 → 费用归属A
2026-04: 张三调动到结算主体B → 3月费用仍归属A
2026-04: 张三在结算主体B新参保 → 4月费用归属B
```

### 退费/补收补退

退费/补收补退按原费用产生时的归属处理：

```
原费用：2026-03 归属A的5000元
退费：2026-05 产生退费 → 退费归属A
```

### 合并付款

合并付款申请以发起的结算主体为财务归属，内部划转费用通过内部结算追溯：

```
结算主体X发起合并付款100,000元：
  - 本主体费用：60,000元（归属X）
  - Y主体费用：30,000元（内部划转，归属X）
  - Z主体费用：10,000元（内部划转，归属X）
```

## 数据迁移注意事项

1. **迁移前备份**：全量备份相关表数据
2. **迁移窗口**：建议在非工作时间执行
3. **迁移验证**：迁移后抽样验证版本记录与历史异动的一致性
4. **回滚预案**：保留回滚脚本，出现问题时可快速恢复

## 风险评估

| 风险 | 影响 | 应对 |
|------|------|------|
| 历史数据不完整 | 部分员工版本记录缺失 | 迁移脚本增加兜底逻辑，用当前属性补全 |
| 费用表数据量大 | ALTER TABLE 耗时较长 | 提前评估表大小，必要时在低峰期执行 |
| 新旧数据并存 | 迁移期间查询逻辑需兼容 | 迁移期间前端增加「员工归属」筛选条件，新旧查询并存 |
